## 2.0.0 - 2026-04-10
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, String, ...)` overloads have been removed from `LabReportClient`, `AsyncLabReportClient`, and their raw counterparts. Callers must switch to the `File`-based `parserCreateJob(File, BodyCreateLabReportParserJob)` overloads instead.
* The `INSULIN_INJECTION` constant has been removed from the `IntervalTimeseriesExprTimeseries` enum. Code that references this constant or includes it in a switch statement must be updated.
* New in this release: SDK request/response logging can now be enabled by passing a `LogConfig` to `ClientOptions.Builder.logging(LogConfig)`. Logging is silent by default. Additionally, SSE streams now support event-level discrimination via the new `Stream.fromSseWithEventDiscrimination(...)` factory methods.
* Several breaking changes have been made to lab test and link APIs:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field type has changed from `String` to the `AppointmentPscLabs` enum. Callers must replace raw string values with the corresponding `AppointmentPscLabs` constant.
* **`BulkExportConnectionsBody`**: The `teamId` field and its builder methods (`teamId(...)`) have been removed. Remove any calls to `.teamId(...)` in your builder chains.
* **`RawLabReportClient`**: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Migrate to the `File`-based `parserCreateJob(File, ...)` overloads instead.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* The `teamId` field (and its associated `getTeamId()` / `teamId(...)` builder methods) has been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers that set or read `teamId` must remove those references.
* `UserInfoCreateRequest.getAddress()` now returns `UserAddress` instead of `Address`, and the `AddressStage.address(...)` builder method now accepts `UserAddress`. Update any code passing or receiving an `Address` object in this context to use `UserAddress`.
* New optional fields have also been added: `ManualConnectionData` gains `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` gains `accessNotes`.
* The `AggregateExprArg.Visitor` interface now requires three additional `visit` method implementations: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing class implementing `Visitor` must be updated to add these methods or it will fail to compile. Additionally, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields, and `AppointmentRescheduleRequest` exposes an optional `appointmentNotes` field. New enums `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm` are also available.
* Several breaking changes have been made to order and lab result types:
* **`ClientFacingOrder`**: The `getParentId()` method and corresponding `parentId` builder stage have been removed. The staged builder now requires a new mandatory `lastEvent` step (between `updatedAt` and `hasAbn`). Two new fields are available: `clinicalNotes` (optional) and `lastEvent` (required, the latest lifecycle event).
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` (an enum) instead of `String`. Callers must be updated to handle the enum type.
* **`HistoricalPullCompleted`**: The `isFinal` parameter has been removed from the builder (the `IsFinalStage` interface is gone and `getIsFinal()` now always returns `true`). Update any builder chains that called `.isFinal(...)`.
* **`ClientFacingOrderEvent`**: A new optional `statusDetail` field of type `OrderStatusDetail` has been added.
* **`LabResultsRaw`**: A new optional `orderTransaction` field of type `ClientFacingOrderTransaction` has been added.
* A new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been introduced.
* Several breaking changes have been made to model types in this release:
* **`ParsingJob`**: The `jobId` field and `getJobId()` method have been removed. The `JobIdStage` builder interface has been removed; callers using `.id(...).jobId(...)` must update their builder chains to `.id(...)` (which now advances directly to `StatusStage`).
* **`QueryGroupByItem.Visitor`**: Three new `visit` methods have been added (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). Any exhaustive `Visitor` implementation must be updated to implement these new methods.
* **`Placeholder`**: The `PlaceholderStage` interface and the required `placeholder(boolean)` builder step have been removed. Callers must update builder usage accordingly; `getPlaceholder()` now always returns `true`.
* New optional `accessNotes` field has been added to `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* Several breaking changes have been made to the SDK's type model:
* **`UserInfo.address`** now accepts and returns `UserAddress` instead of `Address`. Callers must update references and builder calls accordingly.
* **`UserRefreshSuccessResponse` and `UserRefreshErrorResponse`**: The `SuccessStage` builder interface and the `success(boolean)` builder method have been removed. The `builder()` method now returns `UserIdStage` directly. The `getSuccess()` method now returns a hardcoded `Boolean` constant rather than a stored field. Update any code that sets `success` via the builder.
* **`QuerySelectItem.Visitor`** has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing `Visitor` implementation must be updated to implement these methods.
* `LinkBulkImportRequestTeamId` and `LinkListBulkOpsRequestTeamId` have been removed and replaced by `ResultMetadataGender` and `SearchMode` respectively.
* New additions: `UsAddress` now supports an optional `accessNotes` field, and new enum types `ResultMetadataGender` and `SearchMode` are available in `com.vital.api.types`.
* The `getMarkers` method on `LabTestsClient` now accepts a `lab_slug` filter via `LabTestsGetMarkersRequest`, allowing callers to scope results to a specific lab. Javadoc on `getMarkers` and `getMarkersForLabTest` overloads has also been updated to clarify that `getMarkers` excludes reflex markers while `getMarkersForLabTest` includes them.
* The `getMarkers` method now supports an additional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing consumers to filter markers by lab slug. Javadoc for `getMarkers` and `getMarkersForLabTest` has also been updated to clarify which markers are included in each response.
* The link provider manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, enabling mobile SDK version tracking. The `team_id` query parameter has been removed from several bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`, and the paginated list endpoint).
* The SDK now supports configurable logging via the new `LogConfig` class. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response logging with control over log level, output destination, and silent mode.
* Two new top-level resource clients are now available: `vital.compendium()` (`CompendiumClient`) and `vital.labAccount()` (`LabAccountClient`).
* Several enums have been expanded: `AppointmentProvider`, `ClientFacingLabs`, and `Labs` now include `SONORA_QUEST`; `Labs` also adds `CRL`; `ManualProviders` adds `SAMSUNG_HEALTH`; `OrderLowLevelStatus` adds `CORRECTED` and `LAB_PROCESSING_BLOCKED`; `OrderStatus` adds `SAMPLE_WITH_LAB_TESTKIT_LAB_PROCESSING_BLOCKED` and four new `CORRECTED` terminal states; `PasswordProviders` adds `TANDEM_SOURCE`; and `PayorCodeExternalProvider` adds `APERO` and `PVERIFY`.
* The SDK now includes a new `CompendiumClient` (and its async counterpart `AsyncCompendiumClient`) with `search()` and `convert()` methods for interacting with the Compendium API. A new `SseEventParser` utility class has been added to support parsing Server-Sent Events with discriminated union types. Builder classes for several request types now expose `additionalProperty` and `additionalProperties` methods to pass arbitrary extra fields.
* The SDK now includes a new `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request body classes — `SearchCompendiumBody` and `ConvertCompendiumBody` — are available for structured compendium search and conversion operations. Several existing request builder classes also gain `additionalProperty` / `additionalProperties` methods for attaching custom key-value pairs to requests.
* The SDK now includes a `RawLabAccountClient` for accessing team lab accounts with raw HTTP response details. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`, and all lab test request builders now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* Request builder classes across all SDK resources now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary fields to request objects, supporting custom or undocumented API parameters without requiring SDK updates.
* All request builder classes (under `user` and `vitals` resources) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra properties to requests, providing forward compatibility with new API parameters without requiring a full SDK update.
* The builder for all Vitals request classes (e.g. `VitalsHeartrateRequest`, `VitalsGlucoseRequest`, `VitalsCholesterolLdlRequest`, and many more) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface. These new methods allow consumers to attach arbitrary additional properties to any vitals request during construction.
* The builder for vitals request classes (e.g. `VitalsStepsRequest`, `VitalsRespiratoryRateGroupedRequest`, and others) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on both the `_FinalStage` interface and `Builder`. These new optional builder methods allow consumers to attach arbitrary additional properties to vitals request objects.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on a large number of request and type builder classes, allowing consumers to attach arbitrary extra properties to requests and objects. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for working with lab test canonical matching results, exposing fields such as LOINC set hash, display name, aliases, LOINC/CPT codes, popularity score, and confidence. `ClientFacingAppointment` gains a new optional `appointmentNotes` field. Additionally, all affected builder types now expose `additionalProperty` and `additionalProperties` convenience methods for setting arbitrary extra properties.
* Builder classes for `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields when constructing these objects.
* The SDK now supports setting arbitrary additional properties on builder objects for over 30 `ClientFacing*` types via new `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* The SDK now supports attaching arbitrary additional properties when building `ClientFacing*` model objects via new `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` builder methods across over 25 types. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* Builder instances for `ClientFacing*` types (including `ClientFacingHeartRateTimeseries`, `ClientFacingHrvTimeseries`, `ClientFacingInsulinInjection*`, `ClientFacingInsurance`, `ClientFacingLab`, and many others) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on their `_FinalStage` interface, allowing consumers to attach arbitrary extra properties when constructing these objects. Two new enums, `ClientFacingInsulinInjectionSampleBolusPurpose` and `ClientFacingInsulinInjectionSampleDeliveryMode`, have also been added to support insulin injection sample metadata.
* The SDK now includes three new public types: `ClientFacingLabAccount` (representing a lab account with billing and allowlist metadata), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (webhook event types for lab report parsing jobs). Several existing builder types also gained new `additionalProperty` and `additionalProperties` builder methods for passing through arbitrary extra fields.
* Builder objects for numerous `ClientFacing*` types (including `ClientFacingMenstrualCycle`, `ClientFacingProfile`, `ClientFacingPayor`, `ClientFacingProvider`, and many others) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it straightforward to attach arbitrary extra fields when constructing these objects programmatically. Additionally, the `ClientFacingMenstrualCycleSourceType` enum has a new `INSULIN_PUMP` value.
* The `ClientFacingSleep` type now includes a new optional `recoveryReadinessScore` field (0–100) representing the provider's recovery/readiness proxy, sourced from Oura readiness score, Whoop recovery score, and Ultrahuman recovery. A new `INSULIN_PUMP` value has been added to the `ClientFacingSleepCycleSourceType` enum. Additionally, all affected `ClientFacing*` builder types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods for attaching arbitrary extra properties when constructing objects.
* The builders for numerous `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and concrete `Builder` classes. This allows SDK consumers to attach arbitrary extra properties to model objects directly through the fluent builder API without requiring subclassing or other workarounds.
* The builder for numerous `ClientFacing*` model types (including workout, water, weight, wheelchair push, sleep, and related event/sample/historical-pull classes) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new optional builder methods allow SDK consumers to attach arbitrary extra properties when constructing model objects.
* The SDK now includes three new types: `CompendiumSearchLabs` (an enum for lab identifiers), `ConvertCompendiumResponse` (a response model for compendium conversions), and `DerivedReadinessColumnExpr` with its companion `DerivedReadinessColumnExprDerivedReadiness` enum. Builder interfaces across many existing types have been extended with new `additionalProperty` and `additionalProperties` methods, allowing consumers to attach arbitrary metadata when constructing objects.
* Builder instances for many SDK model types (e.g. `DiscreteTimeseriesExpr`, `Energy`, `Fats`, `GetMarkersResponse`, `GetOrdersResponse`, all `Grouped*` timeseries response types, and more) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, making it easier to pass custom fields through the builder pattern. A new `GetTeamLabAccountsResponse` model class has also been added, containing an optional list of `ClientFacingLabAccount` objects.
* The builder classes for all `Grouped*` and `Grouped*Response` types (e.g. `GroupedHeartRate`, `GroupedGlucoseResponse`) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary extra fields when constructing these objects, improving forward-compatibility with evolving API responses.
* The builders for all `Grouped*` and `Grouped*Response` types (e.g., `GroupedIge`, `GroupedIgg`, `GroupedSteps`, `GroupedVo2MaxResponse`, etc.) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary metadata when constructing these objects.
* The SDK now supports setting arbitrary additional properties on builder objects for grouped timeseries and lab-related types via new `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on their `_FinalStage` interfaces and `Builder` classes. Two new types have been added: `InsulinInjectionTimeseriesExpr` (with its accompanying `InsulinInjectionTimeseriesExprField` enum) for insulin injection timeseries query expressions, and `LabAccountDelegatedFlow` enum for lab account delegation flow states.
