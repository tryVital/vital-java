## 2.0.0 - 2026-04-02
* **Breaking changes:**
* The `parserCreateJob(InputStream, String)`, `parserCreateJob(InputStream, String, MediaType)`, and their `RequestOptions`-bearing variants have been removed from `LabReportClient` and `AsyncLabReportClient`. Migrate to the `File`-based overload: `parserCreateJob(File file, BodyCreateLabReportParserJob request)`.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum. Update any switch statements or comparisons that reference this value.
**New features:**
* Configurable SDK request/response logging is now available. Pass a `LogConfig` instance to `ClientOptions.Builder#logging(LogConfig)` to enable log output (silent by default).
* `Stream` now supports SSE streams with event-level discrimination for discriminated-union payloads via the new `Stream.fromSseWithEventDiscrimination(...)` factory methods.
* Several breaking changes have been made to lab test and link request models:
* The `parserCreateJob(InputStream, ...)` overloads on `RawLabReportClient` have been removed; use the `File`-based overloads instead.
* The `lab` field in `LabTestsGetPscAppointmentAvailabilityRequest` now requires an `AppointmentPscLabs` enum value instead of a raw `String`; update builder calls accordingly.
* The `teamId` field has been removed from `BulkExportConnectionsBody`; remove any `teamId(...)` calls on its builder.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to bulk-operation and user request types:
* The `getTeamId()` method and `teamId(...)` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any calls to these methods.
* The `address` field in `UserInfoCreateRequest` now uses `UserAddress` instead of `Address`. Update any `.address(Address)` calls to pass a `UserAddress` instance instead.
* New optional fields have also been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* The `AggregateExprArg.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing `Visitor` must now implement these methods or a compilation error will occur.
* In addition, the SDK introduces new optional fields on `ClientFacingInsulinInjectionSample` (`deliveryMode`, `deliveryForm`, `bolusPurpose`) and `AppointmentRescheduleRequest` (`appointmentNotes`), along with new enums `AppointmentPscLabs`, `ClientFacingInsulinInjectionSampleDeliveryForm`, `ClientFacingInsulinInjectionSampleDeliveryMode`, and `ClientFacingInsulinInjectionSampleBolusPurpose`.
**Migration:** Add the three new `visit(...)` method implementations to any class that implements `AggregateExprArg.Visitor`.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added, inserting a new `LastEventStage` step into the builder chain between `updatedAt` and `hasAbn`. A new optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The `status` field and its builder method have changed from `String` to the new `OrderTransactionStatus` enum type. Update all `StatusStage.status(...)` calls accordingly.
* **`HistoricalPullCompleted`**: The `isFinal` field has been removed from the model and its builder. The `IsFinalStage` interface no longer exists; remove any `.isFinal(...)` calls from builder chains. `getIsFinal()` now always returns `true`.
* New additions (backward-compatible): `ClientFacingOrderEvent` gains an optional `statusDetail` field; `LabResultsRaw` gains an optional `orderTransaction` field; a new `LabAccountStatus` enum is available.
* **Breaking changes in this release:**
* `ParsingJob`: The `jobId` field (`getJobId()`) and its associated `JobIdStage` builder step have been removed. Remove any calls to `getJobId()` and drop the `.jobId(...)` step from builder chains — `id()` now advances directly to `StatusStage`.
* `QueryGroupByItem.Visitor<T>`: Three new `visit` methods are required — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. All implementations of this interface must add these methods or compilation will fail.
* `Placeholder`: The staged builder (`PlaceholderStage` / `placeholder(boolean)`) has been removed. Use `Placeholder.builder().build()` directly — the type is now a stateless marker.
**New capabilities:**
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now expose an optional `accessNotes` field (`getAccessNotes()`) and the corresponding builder methods `accessNotes(String)` / `accessNotes(Optional<String>)`.
* The `getMarkers` method now supports filtering by lab slug via a new optional `labSlug` field on `LabTestsGetMarkersRequest`. The Javadoc for `getMarkers` has been updated to clarify that only active, orderable markers are returned and reflex markers are excluded. All overloads of `getMarkersForLabTest` now include Javadoc noting that reflex markers are included in the response.
* The `getMarkers` method now supports filtering by `lab_slug` via `LabTestsGetMarkersRequest`. The `getMarkersForLabTest` methods have received updated documentation clarifying that reflex markers are included in the response.
* The `connectManualProvider` method on `LinkClient` now supports sending optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers when provided in the request object. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`, and the connections listing endpoint).
* The SDK now supports configurable HTTP request/response logging via the new `LogConfig` API — silent by default, and opt-in through `VitalBuilder.logging(LogConfig)` or `AsyncVitalBuilder.logging(LogConfig)`. Two new resource clients are available: `vital.compendium()` and `vital.labAccount()` (and their async counterparts). Several enums have been expanded: `Labs` now includes `SONORA_QUEST` and `CRL`; `AppointmentProvider` and `ClientFacingLabs` include `SONORA_QUEST`; `ManualProviders` includes `SAMSUNG_HEALTH`; `OrderLowLevelStatus` includes `CORRECTED` and `LAB_PROCESSING_BLOCKED`; `OrderStatus` includes corrected and lab-processing-blocked statuses for multiple collection types; `PasswordProviders` includes `TANDEM_SOURCE`; and `PayorCodeExternalProvider` includes `APERO` and `PVERIFY`.
* The SDK now includes a new `CompendiumClient` (and async counterpart) with `search` and `convert` methods for interacting with the `v3/compendium` endpoints. A new `SseEventParser` utility class is also available for parsing Server-Sent Events with discriminated union support. Additionally, request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing arbitrary extra properties.
* The SDK now includes a new `LabAccountClient` (and its async counterpart) with a `getTeamLabAccounts()` method for retrieving team lab account information. Two new request body classes, `SearchCompendiumBody` and `ConvertCompendiumBody`, have been added to support compendium search and conversion operations. Several existing request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* The SDK now includes a new `RawLabAccountClient` for retrieving team lab accounts with optional filtering by `lab_account_id` and `status`. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builders now expose `additionalProperty` and `additionalProperties` methods for attaching custom metadata to requests.
* Request builder classes across all resource types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to include arbitrary extra fields in API requests. This affects builders in the `labtests`, `link`, `meal`, `menstrualcycle`, `order`, `payor`, `profile`, `providers`, `sleep`, `sleepcycle`, `team`, `testkit`, and `user` resource packages.
* Request builder classes across the `user` and `vitals` resource packages now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new builder methods allow SDK consumers to attach arbitrary key-value pairs to requests, providing a forward-compatible extension point without requiring SDK changes.
* The SDK now supports passing arbitrary additional properties through the builder for all Vitals request types. New `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods are available on the `_FinalStage` interface and `Builder` implementation of each Vitals request class, enabling callers to attach custom key-value pairs to requests without requiring SDK changes.
* The builder for vitals request types (e.g. `VitalsStepsRequest`, `VitalsWaterRequest`, `VitalsIgeRequest`, and their grouped counterparts) now supports `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary extra properties to requests.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` builder methods on request and type classes, allowing consumers to attach arbitrary extra properties when constructing objects. A new `AwakeningsValueMacroExpr` type is also available, and `AppointmentBookingRequest` gains an optional `appointmentNotes` field.
* The SDK now includes the new `CanonicalCandidate` type for working with lab result canonicalization data, including fields for LOINC/CPT codes, aliases, provider IDs, popularity score, and confidence. `ClientFacingAppointment` gains an optional `appointmentNotes` field. All affected builder types now expose `additionalProperty` and `additionalProperties` methods for setting arbitrary extra fields.
* Builder objects for `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra properties when constructing model instances. These methods are available on both the `_FinalStage` interfaces and their concrete `Builder` implementations.
* Builder classes for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods via the `_FinalStage` interface, allowing consumers to attach arbitrary extra fields when constructing these objects. The `ClientFacingDeviceSourceType` enum also gains a new `INSULIN_PUMP` value.
* Builder classes for numerous `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, making it easier to attach custom metadata when constructing request or response objects. A new `INSULIN_PUMP` source type has also been added to `ClientFacingElectrocardiogramSourceType`.
* The builder for all `ClientFacing*` types now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods on the `_FinalStage` interface, making it easier to attach arbitrary extra fields when constructing these objects. Two new enums have also been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (values: `MEAL`, `CORRECTION`, `MIXED`, `UNKNOWN`) and `ClientFacingInsulinInjectionSampleDeliveryMode` (values: `BASAL`, `BOLUS`).
* The SDK now includes three new types: `ClientFacingLabAccount` for representing lab account data, and `ClientFacingLabReportParsingJobCreatedEvent` / `ClientFacingLabReportParsingJobUpdatedEvent` for handling `lab_report.parsing_job.created` and `lab_report.parsing_job.updated` webhook events respectively. Several existing builder types also now expose `additionalProperty` and `additionalProperties` methods for setting extra fields.
* Builder objects for numerous `ClientFacing` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary metadata when constructing SDK objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingMenstrualCycleSourceType` enum.
* The `ClientFacingSleep` type now exposes a new optional `recoveryReadinessScore` field (a 0–100 integer representing the provider's recovery/readiness proxy, sourced from Oura, Whoop, and Ultrahuman). Builder interfaces across all `ClientFacing*` types now include `additionalProperty` and `additionalProperties` methods for attaching arbitrary metadata. The `ClientFacingSleepCycleSourceType` enum gains a new `INSULIN_PUMP` value.
* Builder classes for `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new methods allow consumers to attach arbitrary extra fields when constructing objects such as `ClientFacingUser`, `ClientFacingTeam`, `ClientFacingStream`, timeseries samples, and webhook event types.
* Builder classes for `ClientFacing*` types (workouts, water, weight, wheelchair push, sleep, walk-in test orders, and others) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. All existing model builder interfaces now expose `additionalProperty` and `additionalProperties` methods, allowing consumers to attach arbitrary extra JSON fields when constructing request or response objects.
