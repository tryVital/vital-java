## 2.0.0 - 2026-04-03
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, String, ...)` method overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers must migrate to the `File`-based overload: `parserCreateJob(File file, BodyCreateLabReportParserJob request)`.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum; any code referencing this constant will fail to compile.
* New in this release: the SDK now supports configurable logging via `ClientOptions.Builder.logging(LogConfig)`, and streaming now supports SSE event-discriminated unions via `Stream.fromSseWithEventDiscrimination()`.
* Several breaking changes have been made to the lab tests and link request models:
* The `parserCreateJob(InputStream, ...)` overloads (four variants) have been removed from `RawLabReportClient`. Use the existing `File`-based `parserCreateJob` overloads instead.
* The `lab` field in `LabTestsGetPscAppointmentAvailabilityRequest` has changed type from `String` to the new `AppointmentPscLabs` enum — update builder calls accordingly.
* The `teamId` field (and its builder methods) have been removed from `BulkExportConnectionsBody`.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* **Breaking changes:**
* The `address` field in `UserInfoCreateRequest` has changed its type from `Address` to `UserAddress`. Callers must update all usages to construct and pass a `UserAddress` instance instead.
* The `getTeamId()` method and `teamId(...)` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any calls to these methods in your code.
**New features:**
* `ManualConnectionData` now supports three new optional fields: `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`.
* `Address` now supports a new optional `accessNotes` field.
* Builder classes across all modified types now expose `additionalProperty(String, Object)` and `additionalProperties(Map)` methods for passing arbitrary extra properties.
* The `AggregateExprArg.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface must be updated to implement these new methods or the code will fail to compile.
* The SDK also adds new optional fields to `ClientFacingInsulinInjectionSample` (`deliveryMode`, `deliveryForm`, `bolusPurpose`) and `AppointmentRescheduleRequest` (`appointmentNotes`), along with new enum types `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm`.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)`, `parentId(Optional<String>)`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) and optional `clinicalNotes` field have been added. The builder stage chain now includes a new `LastEventStage` — callers must call `.lastEvent(...)` after `.updatedAt(...)`.
* **`GetOrderTransactionResponse`**: The `status` field type has changed from `String` to the `OrderTransactionStatus` enum. Update calls to `getStatus()` and builder `.status(...)` accordingly.
* **`HistoricalPullCompleted`**: The `isFinal` field has been removed as a configurable value (it now always returns `true`). The `IsFinalStage` builder interface and `isFinal(boolean)` builder method have been removed — update builder call chains to skip this step.
* **`LinkBulkExportRequestTeamId`** enum has been removed from `com.vital.api.resources.link.types`.
* New additions: `LabResultsRaw` now includes an optional `orderTransaction` field, `ClientFacingOrderEvent` now includes an optional `statusDetail` field, and a new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been added.
* Several breaking changes have been made to the SDK types:
* **`ParsingJob`**: The `jobId` field and `getJobId()` method have been removed. The builder's staged interface has changed — `IdStage` now returns `StatusStage` directly (the intermediate `JobIdStage` is gone). Existing builder chains calling `.id(...).jobId(...).status(...)` must be updated to `.id(...).status(...)`.
* **`Placeholder`**: The `placeholder` boolean field and its required builder stage (`PlaceholderStage`) have been removed. `Placeholder.builder()` now returns a plain `Builder` directly, and `getPlaceholder()` always returns `true`. Callers that used `Placeholder.builder().placeholder(value)` or matched on `PlaceholderStage` must remove that step.
* **`QueryGroupByItem.Visitor`**: Three new visit methods have been added (`visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, `visit(InsulinInjectionTimeseriesExpr)`). Any class implementing `Visitor` must now implement these new methods.
* **`PatientAddress`, `PatientAddressCompatible`, `PatientAddressWithValidation`**: New optional `accessNotes` field added with `getAccessNotes()` and builder methods — fully backward compatible.
* The `UserInfo.getAddress()` method now returns `UserAddress` instead of `Address`; callers must update their type references accordingly. The `UserRefreshErrorResponse` and `UserRefreshSuccessResponse` builders no longer expose a `success(boolean)` builder step — the `SuccessStage` interface has been removed and the builder now starts directly at `UserIdStage`. The `LinkBulkImportRequestTeamId` and `LinkListBulkOpsRequestTeamId` enums have been replaced by `ResultMetadataGender` and `SearchMode` respectively in the `com.vital.api.types` package. Three new `QuerySelectItem` union variants have been added (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`), and existing `Visitor` implementations must be updated to implement the new `visit` methods. `UsAddress` now includes an optional `accessNotes` field.
* The `getMarkers` method now supports filtering by `lab_slug` via `LabTestsGetMarkersRequest`. Javadoc descriptions for `getMarkers` and `getMarkersForLabTest` have been updated to clarify that `getMarkers` excludes reflex markers, while `getMarkersForLabTest` includes them.
* The `getMarkers` method now supports filtering by `lab_slug` via the `LabTestsGetMarkersRequest` builder. The Javadoc for `getMarkers` has been updated to clarify that only active and orderable markers are returned and that reflex markers are excluded. The `getMarkersForLabTest` methods now include Javadoc noting that reflex markers are included in results.
* The link provider manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, populated via new optional fields on the request object. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`, and the paginated listing endpoint).
* The SDK now supports configurable logging via a new `LogConfig` API on `VitalBuilder` and `AsyncVitalBuilder`. Logging is silent by default and can be enabled with a custom log level and optional logger implementation.
* Two new top-level resource clients are now available: `CompendiumClient` (via `vital.compendium()`) and `LabAccountClient` (via `vital.labAccount()`), along with their async counterparts.
* New enum values have been added across several types: `AppointmentProvider.SONORA_QUEST`, `ClientFacingLabs.SONORA_QUEST`, `Labs.SONORA_QUEST` and `Labs.CRL`, `ManualProviders.SAMSUNG_HEALTH`, `OrderLowLevelStatus.CORRECTED` and `LAB_PROCESSING_BLOCKED`, multiple new `OrderStatus` corrected/blocked variants, `PasswordProviders.TANDEM_SOURCE`, and `PayorCodeExternalProvider.APERO` and `PVERIFY`.
* The SDK now includes a new `CompendiumClient` (and async variant) exposing `search()` and `convert()` methods against the v3/compendium API. Several request builder classes (`ActivityGetRequest`, `ActivityGetRawRequest`, `BodyGetRequest`, `BodyGetRawRequest`, `QueryBatch`, `AggregateGetTaskHistoryForContinuousQueryRequest`) now support `additionalProperty()` and `additionalProperties()` for passing arbitrary extra fields. A new `SseEventParser` utility is also available for parsing Server-Sent Events with discriminated-union support.
* The SDK now includes a `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request types, `SearchCompendiumBody` and `ConvertCompendiumBody`, are available for building compendium search and conversion requests. Several existing request builder classes also now expose `additionalProperty` and `additionalProperties` methods for passing through custom fields.
* The SDK now includes a new `RawLabAccountClient` for direct HTTP access to the lab account endpoint (`getTeamLabAccounts`), with support for optional `labAccountId` and `status` filters. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab-test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing arbitrary extra fields.
* Request builder classes across all resource namespaces now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new optional builder methods allow SDK consumers to attach arbitrary extra fields to request payloads, enabling forward-compatible use of undocumented or experimental API parameters.
* All request builder classes in the `user` and `vitals` resource packages now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary additional properties to any request object using the existing fluent builder pattern.
* The builder for all Vitals request classes (e.g. `VitalsCholesterolLdlRequest`, `VitalsGlucoseRequest`, `VitalsHeartrateGroupedRequest`, and many more) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on both the `_FinalStage` interface and the concrete `Builder`. This allows attaching arbitrary extra properties to any Vitals request at build time.
* Vitals request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. This enables consumers to attach arbitrary custom properties to requests across all Vitals request types.
* The SDK now exposes `additionalProperty` and `additionalProperties` builder methods on all request and type builders, allowing consumers to attach arbitrary extra properties. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for lab test canonical matching, with full staged builder support. `ClientFacingAppointment` has a new optional `appointmentNotes` field. All affected builder types now expose `additionalProperty(String, Object)` and `additionalProperties(Map)` convenience methods for setting extra JSON properties.
* Builder classes for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easy to attach arbitrary extra fields when constructing SDK objects. This affects a wide range of types including appointment events, body metrics, blood oxygen/pressure timeseries, basal body temperature samples, and more.

A new `INSULIN_PUMP` value has been added to the `ClientFacingDeviceSourceType` enum.

* Builder objects for `ClientFacing*` types (including `ClientFacingHeartRateAlertChanged`, `ClientFacingHrvTimeseries`, `ClientFacingInsurance`, `ClientFacingLab`, and many more) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields. Two new enums have been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (`MEAL`, `CORRECTION`, `MIXED`, `UNKNOWN`) and `ClientFacingInsulinInjectionSampleDeliveryMode` (`BASAL`, `BOLUS`).
