## 2.0.0 - 2026-04-09
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, ...)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers must migrate to the `parserCreateJob(File, BodyCreateLabReportParserJob)` overload, converting their `InputStream` to a `File` as needed.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum; any references to this constant must be removed.
* New capabilities are also included: `ClientOptions` now supports optional structured logging via a new `Builder.logging(LogConfig)` method (silent by default), and `Stream` gains `fromSseWithEventDiscrimination(...)` factory methods for handling SSE streams backed by discriminated union payloads.
* Several breaking changes have been made to lab test and appointment request types:
* The four `parserCreateJob(InputStream, ...)` overloads on `RawLabReportClient` have been removed. Callers must switch to the `File`-based `parserCreateJob` overloads.
* The `lab` parameter on `LabTestsGetPscAppointmentAvailabilityRequest` (and its builder stage) has changed from `String` to the `AppointmentPscLabs` enum. Update all call sites accordingly.
* The `teamId` field (of type `LinkBulkExportRequestTeamId`) has been removed from `BulkExportConnectionsBody`; remove any calls to `.teamId(...)` on its builder.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request types in the `link` and `user` packages:
* The `getTeamId()` getter and `teamId()` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any calls to these methods and builder setters.
* `UserInfoCreateRequest.getAddress()` now returns `UserAddress` instead of `Address`. Update any code that assigns the result to an `Address`-typed variable.
* New optional fields have also been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* The `AggregateExprArg.Visitor` interface has new required `visit()` overloads for `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`. Any existing implementation of `AggregateExprArg.Visitor` must add these three methods to compile.
* Additionally, the SDK adds optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields to `ClientFacingInsulinInjectionSample`, an optional `appointmentNotes` field to `AppointmentRescheduleRequest`, and two new enums: `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm`.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its getter (`getParentId()`) have been removed. A new required `lastEvent` (`ClientFacingOrderEvent`) field has been added, and the staged builder now requires `lastEvent(...)` to be called after `updatedAt(...)`. New optional `clinicalNotes` field added.
* **`GetOrderTransactionResponse`**: The `status` field type has changed from `String` to the new `OrderTransactionStatus` enum. Callers of `getStatus()` and `Builder.status(...)` must update to use `OrderTransactionStatus`.
* **`HistoricalPullCompleted`**: The `isFinal` builder stage and setter have been removed; `getIsFinal()` now always returns `true`.
* New capabilities: `ClientFacingOrderEvent` gains an optional `statusDetail` field; `LabResultsRaw` gains an optional `orderTransaction` field; a new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) is available.
* **Breaking changes:**
* `ParsingJob`: the `getJobId()` method and the `JobIdStage` builder stage have been removed. Code that calls `parsingJob.getJobId()` or chains `.jobId(...)` in a `ParsingJob.builder()` expression must be updated — the builder now advances directly from `id(...)` to `status(...)`.
* `Placeholder`: the required `placeholder` builder parameter and the `PlaceholderStage` interface have been removed. `Placeholder.builder()` now returns a plain `Builder` directly; `getPlaceholder()` always returns `true`.
**New features:**
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now expose an optional `accessNotes` field via `getAccessNotes()` and the builder methods `accessNotes(String)` / `accessNotes(Optional<String>)`.
* `QueryGroupByItem` adds support for three new expression types: `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`, each with a corresponding `Visitor.visit()` method.
* All affected builder `_FinalStage` interfaces now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` helper methods.
* Several breaking changes have been introduced in this release:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update their code to use `UserAddress`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** builders no longer have a `SuccessStage` or a `success(boolean)` setter — the `success` field is now a hardcoded constant. Any code that calls `.success(...)` on these builders will fail to compile.
* **`QuerySelectItem`** variant type indices have shifted: three new types (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`) were inserted, renumbering all subsequent variants. Implementations of the `Visitor` interface must now also implement `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** enums have been removed and replaced with `ResultMetadataGender` and `SearchMode` respectively.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing callers to scope marker results to a specific lab by slug. Javadoc descriptions for `getMarkers` and `getMarkersForLabTest` have been updated to better reflect their behavior, including reflex marker handling.
* The `getMarkers` method now supports filtering by `lab_slug` via the `LabTestsGetMarkersRequest` builder. Javadoc for `getMarkers` and `getMarkersForLabTest` has been updated to accurately describe which markers are included (active/orderable vs. reflex). A bug in the `drawCompleted` PATCH request has also been fixed to correctly send an empty request body.
* The SDK now sends optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` HTTP headers when calling the link provider/manual endpoint, controlled by new fields on the request object. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`) — callers that previously set `teamId` on those request objects will no longer see it forwarded to the server.
* The SDK now includes two new top-level resource clients: `CompendiumClient` (accessible via `vital.compendium()`) and `LabAccountClient` (accessible via `vital.labAccount()`), along with their async counterparts.
* Configurable logging has been introduced via `VitalBuilder.logging(LogConfig)` and `AsyncVitalBuilder.logging(LogConfig)`. Logging is silent by default and can be enabled with a custom log level and logger implementation using `LogConfig.builder()`.
**Breaking change:** New values have been added to the following enums — `AppointmentProvider` (`SONORA_QUEST`), `ClientFacingLabs` (`SONORA_QUEST`), `Labs` (`SONORA_QUEST`, `CRL`), `ManualProviders` (`SAMSUNG_HEALTH`), `OrderLowLevelStatus` (`CORRECTED`, `LAB_PROCESSING_BLOCKED`), `OrderStatus` (`SAMPLE_WITH_LAB_TESTKIT_LAB_PROCESSING_BLOCKED`, `COMPLETED_WALK_IN_TEST_CORRECTED`, `COMPLETED_AT_HOME_PHLEBOTOMY_CORRECTED`, `COMPLETED_ON_SITE_COLLECTION_CORRECTED`, `COMPLETED_TESTKIT_CORRECTED`), `PasswordProviders` (`TANDEM_SOURCE`), and `PayorCodeExternalProvider` (`APERO`, `PVERIFY`). If you use exhaustive `switch` statements over any of these enums, you must add cases for the new values.
* The SDK now includes a new `CompendiumClient` (and async variant) with `search` and `convert` methods for interacting with the Compendium API endpoints. A new `SseEventParser` utility has been added to the core module to support parsing Server-Sent Events with discriminated union types. Request builder classes across multiple resources now support `additionalProperty` and `additionalProperties` methods for passing arbitrary extra fields.
* New `LabAccountClient` (and async variant) is now available, exposing the `getTeamLabAccounts()` endpoint for retrieving team lab account information. Two new compendium request classes — `SearchCompendiumBody` and `ConvertCompendiumBody` — have been added to support compendium search and conversion operations. Several existing request builder classes also gain `additionalProperty`/`additionalProperties` methods for passing custom properties.
* The SDK now includes a new `RawLabAccountClient` for direct HTTP access to the lab account endpoint (`GET /v3/lab_test/lab_account`), supporting optional filtering by `labAccountId` and `status`. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing through custom key-value pairs.
* Request builder classes across the SDK now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to requests. This is available on all staged and non-staged builders in the `labtests`, `link`, `meal`, `menstrualcycle`, `order`, `payor`, `profile`, `providers`, `sleep`, `sleepcycle`, `team`, `testkit`, and `user` resources.
* Builder classes for user and vitals request types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra properties to requests, providing forward compatibility with new API fields without requiring SDK updates.
* The builder for all `Vitals*Request` classes now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary extra properties to vitals requests. Existing code requires no changes.
* The builder for all Vitals request classes (e.g. `VitalsStepsRequest`, `VitalsRespiratoryRateGroupedRequest`, etc.) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, enabling consumers to attach arbitrary additional properties when building these requests.

A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.

An optional `appointmentNotes` field has been added to `ClientFacingAppointment`.
