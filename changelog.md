## 2.0.0 - 2026-04-08
* **Breaking changes in this release:**
* The `parserCreateJob(InputStream, String)` method and its overloads (accepting `MediaType` and/or `RequestOptions`) have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers should migrate to the `parserCreateJob(File, BodyCreateLabReportParserJob)` variants instead.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum. Any switch statements or references to this constant must be updated.
**New features:**
* The SDK now supports configurable request/response logging. Pass a `LogConfig` instance to `ClientOptions.Builder.logging(LogConfig)` to enable log output; logging is silent by default.
* Several breaking changes have been made to request classes in this release:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field type has changed from `String` to the `AppointmentPscLabs` enum. Callers must replace `lab("LabName")` with the appropriate `AppointmentPscLabs` enum constant.
* **`RawLabReportClient`**: The `parserCreateJob(InputStream, ...)` overloads have been removed. Use the `File`-based `parserCreateJob` overloads instead.
* **`BulkExportConnectionsBody`**: The `teamId` field (of type `LinkBulkExportRequestTeamId`) has been removed. Remove any calls to `.teamId(...)` in your builder usage.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* The `getTeamId()` method and `teamId()` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any calls to these methods in your code.
* `UserInfoCreateRequest.getAddress()` now returns `UserAddress` instead of `Address`, and the `AddressStage.address()` builder method now requires a `UserAddress` argument. Update any code that passes an `Address` to use `UserAddress` instead.
* New optional fields have also been added: `ManualConnectionData` gains `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` gains `accessNotes`.
* The `AggregateExprArg.Visitor<T>` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface must add these three methods or it will fail to compile.
* Additionally, the enums `LinkBulkPauseRequestTeamId` and `LinkBulkTriggerHistoricalPullRequestTeamId` have been removed from the `resources/link/types` package and replaced with `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm` in `types`.
* New optional fields `deliveryMode`, `deliveryForm`, and `bolusPurpose` are now available on `ClientFacingInsulinInjectionSample`, and `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field.
* Several order-related model types have been updated with breaking changes and new fields:
* **`ClientFacingOrder`**: The `parentId` field has been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added to the builder chain between `updatedAt` and `hasAbn`. A new optional `clinicalNotes` field is also available.
* **`GetOrderTransactionResponse`**: The `status` field now returns `OrderTransactionStatus` instead of `String`. Update calls to `getStatus()` and `status(...)` in the builder accordingly.
* **`HistoricalPullCompleted`**: The `isFinal(boolean)` builder stage has been removed; `getIsFinal()` now always returns `true`. Remove the `isFinal` step from any builder chains.
* **`ClientFacingOrderEvent`**: A new optional `statusDetail` field has been added.
* **`LabResultsRaw`**: A new optional `orderTransaction` field has been added.
* A new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been introduced; `LinkBulkExportRequestTeamId` has been removed.
* The `ParsingJob` type no longer exposes a `jobId` field. The builder's `JobIdStage` interface has been removed, and `IdStage.id()` now returns `StatusStage` directly — callers must remove any `.jobId(…)` call from existing builder chains.
* The `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` types now include an optional `accessNotes` field (accessible via `getAccessNotes()`), and the `QueryGroupByItem` union type adds support for three new expression variants: `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`; existing `Visitor` implementations must add handler methods for these new variants.
* Several breaking changes have been made to public API types in this release.
* **`UserInfo`**: The `getAddress()` method now returns `UserAddress` instead of `Address`, and the staged builder's `AddressStage.address()` parameter type has changed to `UserAddress`. Callers must update their code to use `UserAddress`.
* **`UserRefreshSuccessResponse` / `UserRefreshErrorResponse`**: The required `success(boolean)` builder stage (`SuccessStage`) has been removed. The `builder()` method now returns `UserIdStage` directly, and `getSuccess()` returns a hardcoded `Boolean` constant. Any code calling `.success(...)` on these builders will fail to compile.
* **`QuerySelectItem`**: Three new variant types (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`) have been inserted, shifting all existing type discriminant indices. Implementations of the `Visitor` interface must now also implement `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`.
* **Removed enums**: `LinkBulkImportRequestTeamId` and `LinkListBulkOpsRequestTeamId` have been removed and replaced by `ResultMetadataGender` and `SearchMode`.
* **`UsAddress`**: New optional `accessNotes` field added (backward compatible).
* The `getMarkers` method now supports filtering by `lab_slug` via `LabTestsGetMarkersRequest`. Javadoc descriptions on `getMarkers` and `getMarkersForLabTest` overloads have been updated to clarify that `getMarkers` excludes reflex markers while `getMarkersForLabTest` includes them.
* The `getMarkers` method on the lab tests client now accepts a new optional `lab_slug` filter parameter, allowing consumers to filter markers by lab slug. Javadoc has also been added to `getMarkersForLabTest` overloads to clarify that reflex markers are included in those results.
* The SDK now sends optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers when connecting a manual provider via `LinkClient`, allowing the Vital API to identify the originating mobile SDK version. The `team_id` query parameter has also been removed from several bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`).
* The SDK now supports configurable logging via the new `LogConfig` API. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response logging with customizable log levels and logger backends (silent by default).
* Two new resource clients are now available: `CompendiumClient` (via `vital.compendium()`) and `LabAccountClient` (via `vital.labAccount()`).
* Several enums have been expanded: `Labs`, `AppointmentProvider`, and `ClientFacingLabs` now include `SONORA_QUEST`; `Labs` also adds `CRL`; `ManualProviders` adds `SAMSUNG_HEALTH`; `PasswordProviders` adds `TANDEM_SOURCE`; `PayorCodeExternalProvider` adds `APERO` and `PVERIFY`; `OrderLowLevelStatus` adds `CORRECTED` and `LAB_PROCESSING_BLOCKED`; and `OrderStatus` adds five new corrected/blocked status values.
* The SDK now includes a new `CompendiumClient` (and `AsyncCompendiumClient`) with `search()` and `convert()` methods for accessing the v3 compendium API endpoints. A new internal `SseEventParser` utility has been added with support for both data-level and event-level Server-Sent Events discrimination. Request builder classes for Activity, Body, QueryBatch, and Aggregate resources now expose `additionalProperty` and `additionalProperties` methods for attaching arbitrary extra fields to requests.
* The SDK now includes a new `LabAccountClient` (and async variant) for retrieving team lab accounts via the `/v3/lab_test/lab_account` endpoint, with optional filtering by lab account ID and status. Two new compendium request types — `SearchCompendiumBody` and `ConvertCompendiumBody` — are also available for compendium search and conversion operations. Additionally, several existing request builder classes now expose `additionalProperty` and `additionalProperties` methods for attaching custom query parameters.
* The SDK now includes a new `RawLabAccountClient` that exposes a `getTeamLabAccounts()` method for retrieving team lab accounts with optional filtering by `lab_account_id` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`, allowing clinical notes to be included when creating orders. Additionally, all lab test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing through custom properties.
* Request builder classes across all resource types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new builder methods allow SDK consumers to attach arbitrary extra fields to request payloads without waiting for new SDK model properties.
* Builder classes for all `user` and `vitals` request types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary extra properties to request objects, providing forward compatibility with API fields not yet modeled in the SDK.
* The builder for all `Vitals*Request` classes now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary extra properties to requests. Existing code is unaffected.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods on their `_FinalStage` interface and `Builder` implementations, allowing consumers to attach arbitrary metadata to requests. This affects all vitals request types including grouped and non-grouped variants.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on all request and type builder classes, allowing consumers to attach arbitrary extra fields to any object. A new `AwakeningsValueMacroExpr` type has been introduced. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for lab test canonical matching, with fields for LOINC set hash, display name, aliases, LOINC/CPT codes, provider IDs, popularity score, and confidence. An optional `appointmentNotes` field has been added to `ClientFacingAppointment`. Builder classes across many response types now expose `additionalProperty` and `additionalProperties` methods for setting arbitrary extra fields.
* Builder classes for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary extra properties when constructing these objects.
* The builders for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, making it straightforward to attach arbitrary metadata to any request object. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` builder methods on a wide range of `ClientFacing*` types, making it easier to attach arbitrary metadata when constructing objects. A new `INSULIN_PUMP` source type has also been added to `ClientFacingElectrocardiogramSourceType`.
* The builder for numerous `ClientFacing*` types now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on the `_FinalStage` interface, enabling consumers to attach custom metadata when constructing these objects. Two new enums have been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (MEAL, CORRECTION, MIXED, UNKNOWN) and `ClientFacingInsulinInjectionSampleDeliveryMode` (BASAL, BOLUS) for improved insulin injection data support.
* The SDK now includes three new types: `ClientFacingLabAccount` for representing lab account data, `ClientFacingLabReportParsingJobCreatedEvent` and `ClientFacingLabReportParsingJobUpdatedEvent` for handling lab report parsing job lifecycle webhook events. Several existing builder types (`ClientFacingLabLocation`, `ClientFacingLeanBodyMassChanged`, `ClientFacingLeanBodyMassHistoricalPullCompleted`, `ClientFacingLeanBodyMassSample`, `ClientFacingLoinc`, `ClientFacingMarker`, `ClientFacingMarkerComplete`, `ClientFacingMealChanged`) now expose `additionalProperty` and `additionalProperties` methods on their `_FinalStage` interface.
* The SDK now supports setting arbitrary additional properties on builder objects for over 25 `ClientFacing*` types via new `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` builder methods. A new `INSULIN_PUMP` value has also been added to the `ClientFacingMenstrualCycleSourceType` enum.
* The `ClientFacingSleep` type now exposes a `getRecoveryReadinessScore()` field (a 0–100 integer) representing a provider recovery/readiness score, currently sourced from Oura, Whoop, and Ultrahuman. A new `INSULIN_PUMP` value has been added to the `ClientFacingSleepCycleSourceType` enum. Builder classes across many `ClientFacing*` types now support `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods for attaching custom metadata.
* Builder instances for all `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary metadata to these objects when constructing them via the builder API. These methods are also declared on the `_FinalStage` interface for consistent use in staged-builder flows.
* The builder classes for numerous `ClientFacing*` and response types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new methods allow consumers to attach arbitrary extra fields when constructing objects via the fluent builder API.
* The SDK now includes three new types: `ConvertCompendiumResponse` (for lab test compendium conversion results), `CompendiumSearchLabs` (an enum of supported lab providers), and `DerivedReadinessColumnExpr` with its associated `DerivedReadinessColumnExprDerivedReadiness` enum. Additionally, builder classes across many existing types now expose `additionalProperty` and `additionalProperties` methods, making it easier to pass through arbitrary extra fields when constructing request and response objects.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on over 30 type builders, allowing consumers to attach arbitrary extra fields when constructing objects. A new `GetTeamLabAccountsResponse` type has also been added, providing access to a list of `ClientFacingLabAccount` entries via its optional `getData()` method.
* The builder for all `Grouped*` and `Grouped*Response` types (e.g. `GroupedHeartRate`, `GroupedGlucoseResponse`, `GroupedCaloriesBasal`, etc.) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra fields when constructing these objects via the builder pattern.
* The builders for all `Grouped*` and `Grouped*Response` types (e.g. `GroupedSteps`, `GroupedIge`, `GroupedVo2MaxResponse`) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary extra properties when constructing these objects.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on over 25 grouped timeseries, response, and domain types, making it straightforward to attach arbitrary extra fields to objects. Two new public types have been added: `InsulinInjectionTimeseriesExpr` (with its companion `InsulinInjectionTimeseriesExprField` enum) for insulin-injection timeseries query expressions, and `LabAccountDelegatedFlow`, a new enum describing lab-account delegation modes. The `IndexColumnExprIndex` enum also gains a new `DERIVED_READINESS` value.
* The SDK now supports setting arbitrary additional properties directly on builders via new `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, available across more than 20 type classes. A new `OrderStatusDetail` enum has also been added, providing constants for fulfillment and demographic-related order status detail values.
