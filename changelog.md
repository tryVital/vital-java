## 2.0.0 - 2026-04-07
* The `parserCreateJob(InputStream, String)` overloads and their `MediaType`/`RequestOptions` variants have been removed from `LabReportClient`, `AsyncLabReportClient`, and their raw counterparts. Callers must switch to the `File`-based `parserCreateJob(File, BodyCreateLabReportParserJob)` overloads. Additionally, the `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum — any exhaustive switch statements referencing it must be updated.
* New additions in this release: the SDK now supports configurable HTTP request/response logging via `ClientOptions.Builder.logging(LogConfig)` (silent by default), and streaming now supports SSE event-level discrimination through the new `Stream.fromSseWithEventDiscrimination()` factory methods.
* Several breaking changes have been made to lab-related request types:
* The four `parserCreateJob(InputStream, ...)` overloads on `RawLabReportClient` have been removed; use the `File`-based overloads instead.
* The `lab` field in `LabTestsGetPscAppointmentAvailabilityRequest` has changed from `String` to the `AppointmentPscLabs` enum — callers must replace raw string values with the corresponding enum constant.
* The `teamId` field (and its getter/builder methods) has been removed from `BulkExportConnectionsBody`.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* The `teamId` field and its associated builder methods (`teamId(...)`) have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Additionally, `UserInfoCreateRequest.getAddress()` now returns `UserAddress` instead of `Address` — callers must update their type references accordingly. Callers should remove all `.teamId(...)` builder calls and update any `Address` type references in `UserInfoCreateRequest` contexts to `UserAddress`.
* Three new variants (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`) have been added to `AggregateExprArg`. Any class that implements `AggregateExprArg.Visitor` must now implement the corresponding `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)` methods or it will fail to compile. Additionally, `LinkBulkPauseRequestTeamId` and `LinkBulkTriggerHistoricalPullRequestTeamId` have been removed and replaced with `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm` respectively. New optional fields (`deliveryMode`, `deliveryForm`, `bolusPurpose`) are now available on `ClientFacingInsulinInjectionSample`, and `appointmentNotes` is now available on `AppointmentRescheduleRequest`.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(...)`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added as a mandatory builder stage between `updatedAt` and `hasAbn`. An optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` (an enum) instead of `String`. Update builder calls to pass `OrderTransactionStatus` values.
* **`HistoricalPullCompleted`**: The `isFinal` builder stage (`IsFinalStage`) and its setter have been removed; `endDate(...)` now advances directly to `ProviderStage`. `getIsFinal()` now always returns `true`.
* **`ClientFacingOrderEvent`**: A new optional `statusDetail` field has been added.
* **`LabResultsRaw`**: A new optional `orderTransaction` field has been added.
* New `LabAccountStatus` enum introduced with values `ACTIVE`, `PENDING`, and `SUSPENDED`.
* Several breaking changes have been made to existing types in this release.
**`ParsingJob`**: The `jobId` field and `getJobId()` method have been removed. The builder's `JobIdStage` interface no longer exists; builder chains must be updated to go directly from `.id(...)` to `.status(...)`.
**`Placeholder`**: The required `placeholder` boolean field and its associated staged builder interfaces (`PlaceholderStage`, `_FinalStage`) have been removed. `getPlaceholder()` now always returns `true`, and `builder()` returns a plain `Builder` instead of a staged builder.
**`QueryGroupByItem.Visitor`**: Three new `visit()` methods have been added (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). All existing `Visitor` implementations must implement these new methods to compile.
* Additionally, `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now support an optional `accessNotes` field.
* Several breaking changes have been made to the SDK types:
* **`QuerySelectItem.Visitor`** now requires three new `visit()` method implementations: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing class implementing this interface will fail to compile until those methods are added.
* **`UserInfo`**: The `getAddress()` return type and `AddressStage.address()` parameter type have changed from `Address` to `UserAddress`. Callers must update their types accordingly.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`**: The `success` field has been removed from the builder; the `SuccessStage` interface is gone and `success(boolean)` can no longer be called. `getSuccess()` now returns a hardcoded `Boolean` constant.
* **`LinkBulkImportRequestTeamId` and `LinkListBulkOpsRequestTeamId`** have been removed and replaced with `ResultMetadataGender` and `SearchMode` respectively.
* New additions: `UsAddress` now has an optional `accessNotes` field, and the new `ResultMetadataGender` and `SearchMode` enums are available in `com.vital.api.types`.
* The `getMarkers` methods on `LabTestsClient` and `AsyncRawLabTestsClient` now accept an optional `lab_slug` filter via `LabTestsGetMarkersRequest`, allowing callers to narrow marker results to a specific lab by slug. Javadoc has also been added to all `getMarkersForLabTest` overloads, clarifying that reflex markers are included in those results.
* The `getMarkers` method on the lab tests client now supports an optional `lab_slug` query parameter for filtering markers by lab slug. Javadoc descriptions have also been added to `getMarkersForLabTest` overloads and updated on `getMarkers` to clarify that reflex markers are excluded from those results.
* The `LinkClient` now supports sending optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` HTTP headers when calling the link provider manual endpoint, enabling mobile SDK version tracking. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`) — callers relying on this parameter should remove it from their request objects.
* The SDK now supports configurable logging via a new `logging(LogConfig)` method on `VitalBuilder` and `AsyncVitalBuilder`. Logging is silent by default and can be enabled with custom log levels and logger implementations.
* Two new resource clients are now available: `CompendiumClient` (via `vital.compendium()`) and `LabAccountClient` (via `vital.labAccount()`).
* Several enums have been expanded with new values: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `TANDEM_SOURCE` in `PasswordProviders`; `APERO` and `PVERIFY` in `PayorCodeExternalProvider`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; and multiple new `CORRECTED`/`LAB_PROCESSING_BLOCKED` statuses in `OrderStatus`.
* The SDK now includes a new `CompendiumClient` (and async variant) with `search()` and `convert()` methods for interacting with the Compendium API. Several request builder interfaces now expose `additionalProperty()` and `additionalProperties()` methods on their `_FinalStage`, enabling consumers to attach arbitrary properties to requests without rebuilding from scratch. A new `SseEventParser` utility has also been added to the core package to support Server-Sent Events with discriminated union types.
* The SDK now includes a `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request types — `SearchCompendiumBody` and `ConvertCompendiumBody` — are available for searching and converting compendium entries. Several existing request builders also gain `additionalProperty` and `additionalProperties` convenience methods for attaching custom properties.
* The SDK now includes a new `RawLabAccountClient` for retrieving team lab accounts with optional filtering by `lab_account_id` and `status`. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing through custom properties.
* Request builder classes across all resource packages now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary custom fields to request objects. This enables forward-compatible extensibility when working with API endpoints that accept or pass through additional parameters not yet modeled in the SDK.
* Builder classes for request objects across the `user` and `vitals` resources now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, enabling consumers to attach arbitrary metadata to requests without breaking existing builder usage.
* The builder for all Vitals request classes now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary key-value metadata to requests. This is a backward-compatible addition — existing code requires no changes.
* The builders for vitals request types (e.g. `VitalsStepsRequest`, `VitalsRespiratoryRateGroupedRequest`, and all related grouped/ungrouped variants) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface. This allows consumers to attach arbitrary extra properties to requests via the fluent builder API.
* Builder classes across all request and type objects now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now accepts an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for working with lab test canonical matching results, including LOINC/CPT code associations, aliases, and confidence scoring. `ClientFacingAppointment` now exposes an optional `appointmentNotes` field. Builder classes across many response and event types now support `additionalProperty` and `additionalProperties` methods for attaching arbitrary metadata.
* The builder for all `ClientFacing*` model types now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on both the `_FinalStage` interface and the concrete `Builder` class. These new optional builder methods allow consumers to attach arbitrary extra properties to any of these objects without any changes required to existing code.
* Builder interfaces for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, allowing consumers to attach arbitrary metadata when constructing request objects. The `ClientFacingDeviceSourceType` enum also gains a new `INSULIN_PUMP` value.

Two new enums, `ClientFacingInsulinInjectionSampleBolusPurpose` and `ClientFacingInsulinInjectionSampleDeliveryMode`, have been added to support richer insulin injection data modelling.
* The SDK now includes three new types: `ClientFacingLabAccount` (for lab account data), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (for lab report parsing job webhook events). Several existing builder interfaces have been extended with `additionalProperty` and `additionalProperties` methods for greater flexibility when constructing objects.

Additionally, `additionalProperty` and `additionalProperties` builder methods are now available on the `_FinalStage` interface and `Builder` for all affected `ClientFacing*` model types, enabling consumers to attach arbitrary extra properties when constructing objects.
