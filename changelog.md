## 2.0.0 - 2026-04-09
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, String)` overloads (and variants accepting `MediaType` and/or `RequestOptions`) have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers must migrate to the `File`-based overload: `parserCreateJob(File, BodyCreateLabReportParserJob)`.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum. Any code referencing this constant must be updated.
* New in this release: the SDK now supports configurable logging. Pass a `LogConfig` instance to `ClientOptions.Builder.logging(LogConfig)` to enable request/response logging (disabled by default). Additionally, the `Stream` class now supports SSE event-level discrimination via the new `fromSseWithEventDiscrimination` factory methods.
* Several breaking changes have been made to lab test and link request types:
* The `parserCreateJob(InputStream, String)` overloads (and variants accepting `MediaType` and/or `RequestOptions`) have been removed from `RawLabReportClient`. Use the `File`-based `parserCreateJob` overloads instead.
* The `lab` parameter in `LabTestsGetPscAppointmentAvailabilityRequest` has changed type from `String` to `AppointmentPscLabs`. Update builder calls to pass an `AppointmentPscLabs` enum value.
* The `teamId` field and its builder methods have been removed from `BulkExportConnectionsBody`. Remove any usage of `teamId(...)` from your builder chains.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* The `teamId` field (and `getTeamId()` accessor) has been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any `.teamId(...)` builder calls and `getTeamId()` usages from your code.
* The `address` parameter in `UserInfoCreateRequest` has changed from type `Address` to `UserAddress`. Update any call sites that pass an `Address` instance to construct a `UserAddress` instead.
* New optional fields have also been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* The `AggregateExprArg.Visitor<T>` interface now declares three additional abstract methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface must add these three methods to compile successfully.
* Additionally, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields (with corresponding builder methods and new enum types), `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field, and a new `AppointmentPscLabs` enum (`QUEST`, `SONORA_QUEST`) has been introduced.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its getter (`getParentId()`) and builder methods (`parentId(...)`) have been removed. Additionally, a new required `lastEvent` field has been inserted into the staged builder chain — existing builder call chains must now include a `.lastEvent(...)` call after `.updatedAt(...)`.
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` instead of `String`; callers that assigned the result to a `String` variable must be updated.
* **`HistoricalPullCompleted`**: The `isFinal` parameter has been removed from the builder and constructor. The `IsFinalStage` interface no longer exists, and `getIsFinal()` now always returns `true`. Existing staged builder chains that called `.isFinal(...)` must remove that call.
* New additions (backward-compatible): `ClientFacingOrder` now exposes `clinicalNotes` and `lastEvent` fields; `ClientFacingOrderEvent` now exposes an optional `statusDetail` field; `LabResultsRaw` now exposes an optional `orderTransaction` field; and a new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been added.
* The `ParsingJob.getJobId()` method and the `JobIdStage` builder interface have been removed; existing code that calls `.jobId(...)` in the `ParsingJob` builder will no longer compile. The `QueryGroupByItem.Visitor` interface has three new required methods (`visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, `visit(InsulinInjectionTimeseriesExpr)`); any custom `Visitor` implementation must add these methods. The `Placeholder` builder no longer exposes a `PlaceholderStage` interface or a `placeholder(boolean)` setter — migrate by calling `Placeholder.builder().build()` directly. New optional `accessNotes` field is now available on `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* Several breaking changes have been introduced in this release:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update their code to use `UserAddress` where `Address` was previously expected, including the `AddressStage.address(@NotNull UserAddress)` builder parameter.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** builders no longer have a `SuccessStage` or `success(boolean)` setter. Code that calls `.builder().success(...)` must be updated — call `.builder()` directly to start with `UserIdStage`. `getSuccess()` now returns a hard-coded `Boolean` value (`false` / `true` respectively).
* **`QuerySelectItem.Visitor`** has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing `Visitor` must add these methods.
* `LinkBulkImportRequestTeamId` and `LinkListBulkOpsRequestTeamId` have been replaced by `ResultMetadataGender` and `SearchMode` respectively.
* New optional fields and types (`accessNotes` on `UsAddress`, `ResultMetadataGender`, `SearchMode`) are also available.
* The `getMarkers` method on `LabTestsClient` now supports an optional `lab_slug` filter parameter, allowing callers to scope marker results to a specific lab by its slug. Javadoc on `getMarkers` and `getMarkersForLabTest` has also been updated to clarify which markers are included or excluded (e.g., reflex markers).
* The `getMarkers` method on `LabTestsClient` now supports filtering by `lab_slug` via `LabTestsGetMarkersRequest`. Javadoc on `getMarkers` has been updated to clarify that reflex markers are not included, and `getMarkersForLabTest` now documents that reflex markers are included in its results.
* The SDK now supports sending optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` headers when calling the link provider manual endpoint. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`) — callers relying on this parameter should remove it from their request objects.
* The SDK now supports configurable logging via the new `LogConfig`, `LogLevel`, and `ILogger` types. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response logging (silent by default).
* Two new resource clients are now available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async counterparts.
* Several enums have been expanded with new values: `Labs` gains `SONORA_QUEST` and `CRL`; `OrderStatus` gains `SAMPLE_WITH_LAB_TESTKIT_LAB_PROCESSING_BLOCKED` and four `CORRECTED` statuses; `OrderLowLevelStatus` gains `CORRECTED` and `LAB_PROCESSING_BLOCKED`; `AppointmentProvider` and `ClientFacingLabs` gain `SONORA_QUEST`; `ManualProviders` gains `SAMSUNG_HEALTH`; `PasswordProviders` gains `TANDEM_SOURCE`; `PayorCodeExternalProvider` gains `APERO` and `PVERIFY`.
* The SDK now includes a new `CompendiumClient` (and `AsyncCompendiumClient`) with `search()` and `convert()` methods for interacting with the compendium API endpoints. A new `SseEventParser` utility has also been added to the core package to support parsing Server-Sent Events with discriminated union types. Builder classes for several request types (Activity, Body, Aggregate, and Query) now expose `additionalProperty` and `additionalProperties` methods for attaching arbitrary extra fields to requests.
* The SDK now includes a new `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request types, `SearchCompendiumBody` and `ConvertCompendiumBody`, are available for lab test search and conversion operations. Several existing request builder classes also gain `additionalProperty` and `additionalProperties` methods for passing extra parameters.
* The SDK now includes a `RawLabAccountClient` for accessing lab account endpoints with raw HTTP response metadata. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. Request builder classes across the lab tests resource now expose `additionalProperty` and `additionalProperties` helper methods for passing arbitrary extra fields.
* Request builder classes across the SDK now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to any request object. This improves forward compatibility when working with API features not yet reflected in the SDK's typed fields.
* All request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary key-value pairs to any request. This is available on both the intermediate and final stages of the fluent builder chain across all user and vitals request types.
* The builder for all `Vitals*Request` classes now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface. These new methods allow SDK consumers to attach arbitrary extra properties to request objects when using the fluent builder API.
* Vitals request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary key-value pairs to requests for forward compatibility with API fields not yet modeled in the SDK. This applies to all Vitals request types including grouped and ungrouped variants.
* The SDK now exposes `additionalProperty` and `additionalProperties` builder methods on all request and type classes, allowing consumers to attach arbitrary key-value metadata to any object. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for working with LOINC/CPT-based lab test canonicalization data. An optional `appointmentNotes` field has been added to `ClientFacingAppointment`. All builder types now expose `additionalProperty` and `additionalProperties` convenience methods for setting arbitrary extra fields.

This allows SDK consumers to attach arbitrary extra properties when constructing these objects via the builder pattern.

A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.

Two new enum types have also been introduced: `ClientFacingInsulinInjectionSampleBolusPurpose` (MEAL, CORRECTION, MIXED, UNKNOWN) and `ClientFacingInsulinInjectionSampleDeliveryMode` (BASAL, BOLUS).
