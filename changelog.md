## 2.0.0 - 2026-04-02
* Several breaking changes are included in this release:
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum. Any exhaustive switch statements or direct references to this value must be updated.
* All `parserCreateJob(InputStream, ...)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers should migrate to the equivalent `File`-based overloads (e.g., `parserCreateJob(File file, BodyCreateLabReportParserJob request)`).
* New in this release: the SDK now supports configurable logging via `ClientOptions.builder().logging(LogConfig)`, and streaming responses gain event-level discrimination support for discriminated-union SSE payloads via `Stream.fromSseWithEventDiscrimination(...)`.
* Several breaking changes have been made to request model types:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field type has changed from `String` to `AppointmentPscLabs`. Callers must update their builder calls from `.lab("someString")` to `.lab(AppointmentPscLabs.valueOf(...))` or the appropriate enum constant.
* **`BulkExportConnectionsBody`**: The `teamId` field and its builder methods (`teamId(...)`) have been removed. Remove any calls to `.teamId(...)` in your builder chains.
* **`RawLabReportClient`**: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Migrate to the `File`-based `parserCreateJob(File, ...)` variants instead.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request types in this release:
* The `teamId` field and associated getter/builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers using `.teamId(...)` or `.getTeamId()` on these builders must remove those calls.
* The `address` field in `UserInfoCreateRequest` has changed from type `Address` to `UserAddress`. Update all usages of `UserInfoCreateRequest.Builder.address(Address)` to pass a `UserAddress` instance instead.
* New optional fields have also been added: `ManualConnectionData` now accepts `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now accepts an `accessNotes` field.
* The SDK introduces several new capabilities:
* `AggregateExprArg` now supports three new union variants — `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr` — with corresponding `visit()` methods added to the `Visitor` interface. **Existing implementations of `AggregateExprArg.Visitor` must be updated to implement these new methods.**
* `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields with new supporting enums.
* `AppointmentRescheduleRequest` now accepts an optional `appointmentNotes` field via its builder.
* New `AppointmentPscLabs` enum (`QUEST`, `SONORA_QUEST`) and `ClientFacingInsulinInjectionSampleDeliveryForm` enum (`STANDARD`, `EXTENDED`) are available.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)` / `parentId(Optional<String>)`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added — existing builder chains calling `.updatedAt(...)` must now chain `.lastEvent(...)` before `.hasAbn(...)`. A new optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The `status` field type has changed from `String` to the new `OrderTransactionStatus` enum. Callers of `getStatus()` or the `status(String)` builder method must be updated.
* **`HistoricalPullCompleted`**: The `isFinal` constructor/builder parameter has been removed and the `IsFinalStage` builder interface is gone. `getIsFinal()` now always returns `true`. Remove any calls to `.isFinal(...)` in builder chains.
* **`LinkBulkExportRequestTeamId`** enum has been removed. Use the new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) where applicable.
* **`ClientFacingOrderEvent`**: New optional `statusDetail` field added (backward compatible).
* **`LabResultsRaw`**: New optional `orderTransaction` field added (backward compatible).
* **Breaking changes:**
* The `jobId` field has been removed from `ParsingJob`. Calls to `getJobId()` will no longer compile, and the staged builder's `JobIdStage` interface has been removed — callers must update builder chains by removing the `.jobId(...)` step.
* The `Visitor` interface in `QueryGroupByItem` has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing `QueryGroupByItem.Visitor` must add these methods or it will fail to compile.
* `Placeholder.builder()` now returns `Builder` directly instead of `PlaceholderStage`; the `placeholder(boolean)` setter and `PlaceholderStage` / `_FinalStage` interfaces have been removed.
**New features:**
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now include an optional `accessNotes` field, accessible via `getAccessNotes()` and settable in the builder with `.accessNotes(...)`.
* Several breaking changes have been introduced in this release:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update their code to use `UserAddress` and its builder when constructing or consuming `UserInfo` objects.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse` builders** no longer require a `success(boolean)` step. Code that calls `.builder().success(true/false).userId(...)` must be updated to call `.builder().userId(...)` directly. `getSuccess()` now returns a hardcoded `Boolean` constant (`true` for success, `false` for error).
* **`QuerySelectItem.Visitor`** gains three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. All existing `Visitor` implementations must add these methods or they will fail to compile.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** have been removed and replaced by `ResultMetadataGender` and `SearchMode` in the `com.vital.api.types` package.
* Additionally, `UsAddress` now supports an optional `accessNotes` field.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing callers to scope marker results to a specific lab by its slug. Javadoc descriptions for `getMarkers` and `getMarkersForLabTest` have also been updated to clarify that `getMarkers` excludes reflex markers while `getMarkersForLabTest` includes them.
* The `getMarkers` method on `LabTestsClient` now supports an optional `lab_slug` query parameter to filter markers by lab slug. The `getMarkersForLabTest` methods now include Javadoc clarifying that reflex markers are included in results. The `getMarkers` Javadoc has been updated to note that only active and orderable markers are returned and reflex markers are excluded.
* The link provider/manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, configurable via new fields on the request object. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`, and the paginated listing endpoint).
* The SDK now supports configurable request/response logging via a new `LogConfig` API — logging is silent by default and can be enabled through `VitalBuilder.logging(LogConfig)` or `AsyncVitalBuilder.logging(LogConfig)`. Two new resource clients are available: `CompendiumClient` (via `vital.compendium()`) and `LabAccountClient` (via `vital.labAccount()`). Several new enum values have been added across `Labs`, `AppointmentProvider`, `ClientFacingLabs`, `ManualProviders`, `PasswordProviders`, `PayorCodeExternalProvider`, `OrderLowLevelStatus`, and `OrderStatus` to reflect updated API capabilities.
* The SDK now includes a new `CompendiumClient` (and async variant) with `search()` and `convert()` methods for interacting with the v3 compendium endpoints. Request builder classes across activity, body, and aggregate resources now support `additionalProperty` and `additionalProperties` methods for passing arbitrary metadata. A new `SseEventParser` utility has also been added with support for event-level and data-level Server-Sent Events discrimination.
* The SDK now includes a new `LabAccountClient` (and async variant) for retrieving team lab accounts via `getTeamLabAccounts()`. Two new compendium request types are available — `SearchCompendiumBody` and `ConvertCompendiumBody` — enabling structured compendium search and conversion. Several existing request builders also gain `additionalProperty` and `additionalProperties` helper methods for passing extra parameters.
* The SDK now includes a new `RawLabAccountClient` for retrieving team lab accounts with optional filtering by `lab_account_id` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builders now consistently expose `additionalProperty` and `additionalProperties` methods for passing through custom properties.
