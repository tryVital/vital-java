## 2.0.0 - 2026-03-27
* The SDK now supports configurable logging for HTTP requests and responses through the new `LogConfig` system. Enhanced Server-Sent Events (SSE) streaming now includes event-level discrimination support for processing discriminated union payloads in real-time scenarios. The `INSULIN_INJECTION` enum value has been removed from `IntervalTimeseriesExprTimeseries`, and several InputStream-based file upload methods have been removed from `LabReportClient` in favor of File-based uploads.
* The SDK removes deprecated InputStream-based file upload methods from RawLabReportClient. Use the File-based upload methods instead. New optional lab configuration fields are now available: labAccountId and labSlug for lab test creation, labSlug for marker filtering, and appointmentNotes for appointment requests.
* The SDK now supports additional optional fields for enhanced data tracking including SDK version information, permission management, address access notes, and appointment notes. New expression types are available for derived readiness columns and insulin injection timeseries data.
* The SDK now supports enhanced insulin injection tracking with optional delivery metadata fields (deliveryMode, deliveryForm, bolusPurpose) and clinical notes on orders. Order events now include optional status details, and lab results can include order transaction information.
* The `ParsingJob.getJobId()` method has been removed. The `jobId` field is no longer available on parsing job objects. Patient address types now include an optional `accessNotes` field for additional delivery instructions.
* New optional access notes field added to US address types for delivery instructions. User address field now uses more specific UserAddress type instead of generic Address type.
* The `getMarkers()` method now supports an optional `lab_slug` parameter for more flexible lab filtering. Method documentation has been updated to clarify that only active and orderable markers are returned, excluding reflex markers.
* The SDK now supports filtering markers by lab using the new `lab_slug` parameter in the `getMarkers` method. Mobile SDK integrations can now pass version information through `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` headers when creating manual provider connections.
* The SDK now includes configurable logging infrastructure with support for request/response logging and custom log levels. New lab provider enum values have been added including Sonora Quest, CRL, Samsung Health, and additional order status values. Two new client resources are available: Compendium and LabAccount for expanded functionality.
* The SDK now includes a new Compendium API client with search and convert functionality. A new SseEventParser utility has been added for handling Server-Sent Events with discriminated union support. Request builders across multiple resource clients now support additional properties via additionalProperty() and additionalProperties() methods.
* New compendium search and conversion capabilities are now available, allowing you to search medical test compendiums and convert between different lab formats. Added lab account management with the `LabAccountClient` class, which provides methods to retrieve and manage team lab accounts. Enhanced all request builders with additional property support for extensibility.
* New lab account management capabilities are now available through the `RawLabAccountClient.getTeamLabAccounts()` method. Additionally, lab test orders can now include clinical notes via the optional `clinicalNotes` parameter in `CreateOrderRequestCompatible`.
* The SDK now supports setting additional properties on all request objects through new builder methods. Use `additionalProperty(key, value)` to set individual properties or `additionalProperties(map)` to set multiple properties at once.
* The SDK now supports setting additional custom properties on all request objects through new builder methods. Use `additionalProperty(key, value)` to set individual properties or `additionalProperties(map)` to set multiple properties at once.
* The SDK now supports adding custom additional properties to all vitals request builders via new `additionalProperty()` and `additionalProperties()` methods. This enables greater flexibility when working with the Vitals API.
* New `additionalProperty()` and `additionalProperties()` methods available on all vitals request builders for passing custom parameters.
* New fluent API methods for setting additional properties on request builders. The `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods are now available on all request builder final stages, enabling custom key-value pairs for extended functionality.
* New `CanonicalCandidate` class available for biomarker canonicalization operations. All model builders now support `additionalProperty()` and `additionalProperties()` methods for setting custom key-value pairs.
* New optional appointment notes field added to appointment records. The SDK now supports retrieving additional notes or instructions provided by appointment providers through the `appointmentNotes` field on `ClientFacingAppointment` objects.
* The SDK now supports setting custom additional properties on data type builders. New `additionalProperty()` and `additionalProperties()` methods are available on all builder classes for enhanced flexibility with dynamic data structures.
* The SDK now supports insulin pump as a device source type. New builder methods `additionalProperty()` and `additionalProperties()` are available on model classes for setting custom properties.
* The SDK now supports setting additional properties on client-facing data objects through new builder methods. Use `additionalProperty(key, value)` to set individual properties or `additionalProperties(map)` to set multiple properties at once.
* New support for insulin injection data modeling with three new enums: ClientFacingInsulinInjectionSampleBolusPurpose, ClientFacingInsulinInjectionSampleDeliveryForm, and ClientFacingInsulinInjectionSampleDeliveryMode. New ClientFacingLabAccount type for managing lab account configurations. New lab report parsing job events: ClientFacingLabReportParsingJobCreatedEvent and ClientFacingLabReportParsingJobUpdatedEvent for tracking parsing operations.
* The SDK now supports setting additional properties on model classes through new `additionalProperty()` and `additionalProperties()` builder methods. A new `INSULIN_PUMP` source type is available for menstrual cycle data collection.
* The SDK now supports setting additional properties on data objects through new builder methods. All client-facing data types now include `additionalProperty()` and `additionalProperties()` methods in their builders, allowing developers to add custom key-value pairs when constructing objects. Additionally, a new `INSULIN_PUMP` source type has been added to sleep cycle data.
* The SDK now includes convenient builder methods for setting additional properties on all model objects. Use `.additionalProperty(key, value)` or `.additionalProperties(map)` when building objects to include custom key-value pairs.

Added compendium search functionality with support for LabCorp, Quest, BioReference, and Sonora Quest laboratories. New lab test conversion capabilities are available through the ConvertCompendiumResponse class.
* New builder methods `additionalProperty()` and `additionalProperties()` are now available on response object builders, allowing better handling of dynamic API fields. A new `GetTeamLabAccountsResponse` class has been added for team lab account operations.
* The SDK now supports setting additional properties on grouped data types through new builder methods. Use `additionalProperty(key, value)` or `additionalProperties(map)` when building grouped response objects.

* New `PerLabCandidate` type available for representing lab test candidate information with marker details, LOINC codes, CPT codes, and confidence scoring.
* The SDK now supports TANDEM_SOURCE and SAMSUNG_HEALTH providers. New data types include ProviderIdConversionResponse for provider ID mapping and RelatedCandidate for canonical data relationships. ResultMetadata now includes an optional gender field with support for male, female, and other values.
