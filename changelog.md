## 2.0.0 - 2026-04-13
* Several breaking changes have been made in this release:
* The `parserCreateJob(InputStream, String[, MediaType][, RequestOptions])` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Replace these calls with the `parserCreateJob(File, BodyCreateLabReportParserJob)` overload, writing the stream contents to a temporary `File` if needed.
* The `INSULIN_INJECTION` enum value has been removed from `IntervalTimeseriesExprTimeseries`. Update any exhaustive switch statements or direct references to this value.
* New capabilities added: SDK HTTP logging can now be configured via `ClientOptions.builder().logging(LogConfig)` (silent by default), and SSE streams now support event-level discrimination via `Stream.fromSseWithEventDiscrimination()`.
* Several breaking changes and new capabilities have been introduced across lab report, lab test, and link request models:
**Breaking changes:**
* `LabTestsGetPscAppointmentAvailabilityRequest`: The `lab` field type has changed from `String` to `AppointmentPscLabs`. Update all call sites to pass an `AppointmentPscLabs` enum value.
* `BulkExportConnectionsBody`: The `teamId` field and associated builder methods (`teamId(...)`, `getTeamId()`) have been removed. Remove any usage from your code.
* `RawLabReportClient`: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Use the `File`-based overloads instead.
**New capabilities:**
* `CreateLabTestRequest` now supports optional `labAccountId` and `labSlug` fields for associating a lab test with a specific lab account or lab slug.
* `LabTestsGetMarkersRequest` now supports an optional `labSlug` field to filter markers by lab slug (takes precedence over `lab_id` if both are provided).
* `RequestAppointmentRequest` now supports an optional `appointmentNotes` field.
* Several breaking changes have been made to request types in the Link and User resources:
* **`BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`** no longer include a `teamId` field. The `getTeamId()` getter and all `teamId(...)` builder methods have been removed. Remove any references to these methods in your code.
* **`UserInfoCreateRequest.getAddress()`** now returns `UserAddress` instead of `Address`, and the `address(...)` builder stage now accepts `UserAddress`. Update any code that uses the return value or passes an `Address` instance to the builder.
* New optional fields have also been added: `ManualConnectionData` gains `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` gains `accessNotes`. All modified builders now expose `additionalProperty` and `additionalProperties` helper methods.
* The `AggregateExprArg.Visitor` interface now requires three additional `visit()` methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Existing implementations of this interface must be updated to add these three methods or compilation will fail.
* The SDK also adds optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields to `ClientFacingInsulinInjectionSample`; an optional `appointmentNotes` field to `AppointmentRescheduleRequest`; and new enums `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm`.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `getParentId()` method and corresponding `parentId(...)` builder methods have been removed. A new required `lastEvent` builder stage has been added between `updatedAt(...)` and `hasAbn(...)`, and new optional `clinicalNotes` field has been added.
* **`GetOrderTransactionResponse`**: `getStatus()` now returns `OrderTransactionStatus` (an enum) instead of `String`; update call sites accordingly.
* **`HistoricalPullCompleted`**: The `isFinal(boolean)` builder step (`IsFinalStage`) has been removed; builder chains should proceed directly from `endDate(...)` to `provider(...)`. `getIsFinal()` now always returns `true`.
* New additions include an optional `statusDetail` field on `ClientFacingOrderEvent`, an optional `orderTransaction` field on `LabResultsRaw`, and a new `LabAccountStatus` enum.
* Several breaking changes have been made to the SDK:
* **`ParsingJob`**: The `jobId` field and its getter `getJobId()` have been removed. Additionally, the `JobIdStage` step has been removed from the staged builder — update builder chains by removing the `.jobId(...)` call (`.id(...)` now advances directly to `StatusStage`).
* **`Placeholder`**: The required `placeholder` builder stage has been removed. The `getPlaceholder()` method now always returns `true`. Update any builder code that called `.placeholder(...)` by removing that call.
* **`QueryGroupByItem.Visitor<T>`**: Three new `visit()` methods have been added to the `Visitor` interface — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing this interface must add implementations for these new methods.
* New additions (backward-compatible):
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now include an optional `accessNotes` field accessible via `getAccessNotes()` and settable through the builder.
* `QueryGroupByItem` now supports `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr` expression types.
* Several breaking changes have been introduced in this release:
* **`QuerySelectItem.Visitor<T>`** now requires three new methods — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)` — which must be implemented by all existing `Visitor` implementations or they will fail to compile.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** have been removed and replaced by `ResultMetadataGender` and `SearchMode` respectively; update all references accordingly.
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`; update call sites and builder usages to use `UserAddress`.
* The required `success(boolean)` builder stage has been removed from `UserRefreshErrorResponse` and `UserRefreshSuccessResponse`; `getSuccess()` now returns a hardcoded `Boolean` constant (`false` and `true` respectively), so builder calls that set `success` must be removed.
* New additions include an optional `accessNotes` field on `UsAddress`, new `additionalProperty`/`additionalProperties` builder methods on several types, and the new `ResultMetadataGender` and `SearchMode` enum types.
* The `getMarkers` method on `LabTestsClient` now supports a new `lab_slug` query parameter for filtering markers by lab slug. The Javadoc for `getMarkers` and `getMarkersForLabTest` methods has also been updated with more descriptive descriptions.
* The `getMarkers` method on `LabTestsClient` now supports an optional `lab_slug` filter parameter, allowing callers to scope marker results to a specific lab by slug. Javadoc has also been added to `getMarkersForLabTest` overloads clarifying that reflex markers are included in those results.
* The link provider manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, allowing mobile SDK version information to be forwarded to the API. Additionally, the `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`).
* The SDK now includes configurable logging support. Use `LogConfig` on `VitalBuilder` or `AsyncVitalBuilder` to enable request/response logging with customizable log levels and logger implementations (silent by default).
* Two new resource clients are now available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async counterparts.
* Several enumerations have been expanded with new values: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `TANDEM_SOURCE` in `PasswordProviders`; `APERO` and `PVERIFY` in `PayorCodeExternalProvider`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; and multiple `CORRECTED` and `LAB_PROCESSING_BLOCKED` status values in `OrderStatus`.
* The SDK now includes a new `CompendiumClient` (and async variant `AsyncCompendiumClient`) with `search()` and `convert()` methods for interacting with the v3/compendium API endpoints. A new `SseEventParser` utility class has also been added to support parsing Server-Sent Events with both event-level and data-level discriminated union patterns. Builder classes for several request types now expose `additionalProperty` and `additionalProperties` methods for passing arbitrary extra fields.
* The SDK now includes a new `LabAccountClient` (and async counterpart) for retrieving team lab accounts via `getTeamLabAccounts()`. Two new compendium request types, `SearchCompendiumBody` and `ConvertCompendiumBody`, are also available to support lab test search and conversion workflows. Additionally, several existing request builder classes now expose `additionalProperty` and `additionalProperties` methods for greater flexibility.
* The SDK now includes a new `RawLabAccountClient` that provides raw HTTP response access to the `getTeamLabAccounts` endpoint, supporting optional filtering by `labAccountId` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builders now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value data.
* All request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to request payloads. These new builder methods are backward compatible and require no changes to existing code.
* All user and vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary additional properties when constructing requests. These methods are available throughout the staged builder chain via the `_FinalStage` interface.
* The builder for all vitals request classes (e.g., `VitalsCholesterolLdlRequest`, `VitalsGlucoseGroupedRequest`, `VitalsHrvRequest`, and many others) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on both the `_FinalStage` interface and the `Builder` implementation. These new optional builder methods allow consumers to attach arbitrary extra properties to request objects.
* The builder for vitals request types (e.g., `VitalsStepsRequest`, `VitalsRespiratoryRateGroupedRequest`, and others) now supports `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface. These new optional methods allow consumers to attach arbitrary key-value pairs to requests when needed.
* Builder objects across all SDK request and type classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, making it easy to attach arbitrary extra fields. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field settable via the builder.
* The SDK now includes a new `CanonicalCandidate` type for lab result canonical matching, exposing fields such as LOINC set hash, display name, aliases, LOINC/CPT codes, provider IDs, popularity score, and confidence. `ClientFacingAppointment` gains a new optional `appointmentNotes` field. All affected builder types now expose `additionalProperty` and `additionalProperties` convenience methods for setting arbitrary extra fields.
* The builder interfaces and implementations for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra properties when constructing these objects.
* Builder classes for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods on both the `_FinalStage` interface and `Builder` implementation, allowing consumers to attach arbitrary metadata when constructing these objects. A new `INSULIN_PUMP` value has been added to the `ClientFacingDeviceSourceType` enum to support insulin pump data sources.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` builder methods on more than 30 `ClientFacing*` types, allowing consumers to attach arbitrary extra fields when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* Builder instances for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary metadata when constructing objects. Two new enums, `ClientFacingInsulinInjectionSampleBolusPurpose` (MEAL, CORRECTION, MIXED, UNKNOWN) and `ClientFacingInsulinInjectionSampleDeliveryMode` (BASAL, BOLUS), have been added to support insulin injection data modeling.
* The SDK now includes three new types: `ClientFacingLabAccount` (for lab account details), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (for lab report parsing job webhook events). Additionally, `additionalProperty` and `additionalProperties` builder methods are now available on the `_FinalStage` interface of several existing response types, including `ClientFacingLabLocation`, `ClientFacingLeanBodyMassSample`, `ClientFacingLoinc`, `ClientFacingMarker`, `ClientFacingMarkerComplete`, and others.
* Builder classes for numerous `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary metadata when constructing SDK objects. The `ClientFacingMenstrualCycleSourceType` enum also gains a new `INSULIN_PUMP` value representing insulin pump devices as a data source.
* The `ClientFacingSleep` type now exposes a new optional `recoveryReadinessScore` field (0–100) representing a provider's recovery/readiness proxy, sourced from Oura readiness score, Whoop recovery score, and Ultrahuman recovery scalar. A new `INSULIN_PUMP` value has been added to the `ClientFacingSleepCycleSourceType` enum. Builder classes across numerous `ClientFacing*` types now support `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods for attaching arbitrary extra data.
* The builders for many `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary extra fields when constructing model objects, providing greater flexibility and forward compatibility with unmodeled API fields.
* Builder classes for numerous `ClientFacing*` and related types (e.g. `ClientFacingWorkout`, `ClientFacingWaterTimeseries`, `ClientSleepResponse`, `ClinicalInformation`) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary extra fields when constructing objects via the builder pattern.
* The SDK now includes three new types: `ConvertCompendiumResponse` (for lab compendium conversion results including source/target lab IDs and provider ID mappings), `CompendiumSearchLabs` (an enum of supported lab slugs), and `DerivedReadinessColumnExpr` with its companion `DerivedReadinessColumnExprDerivedReadiness` enum for readiness column expressions. Builder interfaces across many existing types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, making it easier to attach arbitrary JSON fields when constructing request or response objects.
