## 2.0.0 - 2026-04-13
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, String)` family of overloads (including variants accepting `MediaType` and `RequestOptions`) has been removed from `LabReportClient`, `AsyncLabReportClient`, and their raw counterparts. Callers should migrate to the `File`-based `parserCreateJob(File, BodyCreateLabReportParserJob)` overloads instead.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum; any exhaustive switch statements referencing it must be updated.
* New capabilities also ship in this release: the SDK now supports configurable request/response logging via `LogConfig` (passed to `ClientOptions.builder().logging(...)`), and streaming responses gain a new `SSE_EVENT_DISCRIMINATED` mode for SSE payloads that use the SSE `event` field as a discriminator.
* Several breaking changes and new optional fields have been introduced across lab-related request types:
**Breaking changes:**
* The `parserCreateJob(InputStream, String)`, `parserCreateJob(InputStream, String, MediaType)`, `parserCreateJob(InputStream, String, RequestOptions)`, and `parserCreateJob(InputStream, String, MediaType, RequestOptions)` overloads have been removed from `RawLabReportClient`. Switch to the equivalent `File`-based overloads.
* The `lab` parameter in `LabTestsGetPscAppointmentAvailabilityRequest` has changed from `String` to the `AppointmentPscLabs` enum type. Update builder calls from `.lab("quest")` to `.lab(AppointmentPscLabs.QUEST)` (or the appropriate enum value).
* The `getTeamId()` getter and `teamId(...)` builder methods have been removed from `BulkExportConnectionsBody`. Remove any references to these methods.
**New capabilities:**
* `CreateLabTestRequest` now supports optional `labAccountId` and `labSlug` fields.
* `LabTestsGetMarkersRequest` now supports an optional `labSlug` field to filter markers by lab slug.
* `RequestAppointmentRequest` now supports an optional `appointmentNotes` field.
* Several breaking changes have been made to request and type classes:
* The `teamId` field and its getter/builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers that set or read `teamId` on these classes must remove those references.
* `UserInfoCreateRequest` now accepts a `UserAddress` (instead of `Address`) for the `address` field. Callers must update their `address(...)` builder call to pass a `UserAddress` instance.
* New optional fields have also been added: `ManualConnectionData` gains `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` gains `accessNotes`.
* The `AggregateExprArg.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing class implementing this interface must add these methods to avoid compilation errors.
* Additional additions include optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields on `ClientFacingInsulinInjectionSample`; an optional `appointmentNotes` field on `AppointmentRescheduleRequest`; a new `AppointmentPscLabs` enum; and a new `ClientFacingInsulinInjectionSampleDeliveryForm` enum.
* Several breaking changes have been made to order-related types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`getParentId()`, `parentId(...)`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added; the staged builder now requires a `lastEvent(...)` call between `updatedAt(...)` and `hasAbn(...)`. A new optional `clinicalNotes` field is also available.
* **`GetOrderTransactionResponse`**: The `status` field type has changed from `String` to the new `OrderTransactionStatus` enum. Update calls to `status(...)` and `getStatus()` accordingly.
* **`HistoricalPullCompleted`**: The `isFinal` field and its builder stage have been removed; `getIsFinal()` now always returns `true`. Remove the `isFinal(boolean)` step from any staged builder chains.
* New additions include an optional `statusDetail` field on `ClientFacingOrderEvent`, an optional `orderTransaction` field on `LabResultsRaw`, and a new `LabAccountStatus` enum.
* Several breaking changes have been made to SDK types in this release:
* **`ParsingJob`**: The `jobId` field and `getJobId()` method have been removed. The staged builder no longer includes a `JobIdStage`; calls to `.id(...).jobId(...)` must be updated to `.id(...)` (which now flows directly to `StatusStage`).
* **`Placeholder`**: The required `placeholder` boolean parameter has been removed from the builder. The `PlaceholderStage` and `_FinalStage` interfaces are gone; use the new simplified `Builder` directly via `Placeholder.builder().build()`.
* **`QueryGroupByItem.Visitor`**: Three new `visit` methods have been added (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). Any exhaustive `Visitor` implementation must add these methods.
* New capabilities: the optional `accessNotes` field is now available on `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* The `address` field on `UserInfo` has changed type from `Address` to `UserAddress`. Callers must update their code to use `UserAddress` when constructing or reading `UserInfo` objects.
* The `SuccessStage` builder interface has been removed from `UserRefreshSuccessResponse` and `UserRefreshErrorResponse`; `builder()` now returns `UserIdStage` directly, and `getSuccess()` returns a hardcoded `Boolean` constant instead of accepting a value. Callers that previously called `.success(true/false)` on these builders must remove that call.
* New types `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`, `ResultMetadataGender`, and `SearchMode` are now available. `QuerySelectItem.Visitor` requires implementing three new `visit` methods for these types.
* The `getMarkers` method now supports filtering by `lab_slug` via the `LabTestsGetMarkersRequest` builder. The `getMarkersForLabTest` methods now include Javadoc clarifying that reflex markers are included in those results. Additionally, the `getMarkers` Javadoc now notes that reflex markers are excluded from that endpoint's results.
* The `getMarkers` method on `LabTestsClient` now supports filtering by `lab_slug` via the `LabTestsGetMarkersRequest` builder. Javadoc on `getMarkers` and `getMarkersForLabTest` has been updated to accurately describe which markers are returned (active/orderable vs. all including reflex markers).
* The SDK now sends optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` headers when calling the link provider manual endpoint, enabling mobile SDK version tracking. The `team_id` query parameter has been removed from several bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`).
* The SDK now supports configurable logging via a new `LogConfig` API — pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable request/response logging with customizable log levels and logger implementations. Silent by default.
* Two new resource clients are now available: `CompendiumClient` (sync) / `AsyncCompendiumClient` (async) via `Vital#compendium()`, and `LabAccountClient` (sync) / `AsyncLabAccountClient` (async) via `Vital#labAccount()`.
* New enum values have been added: `SONORA_QUEST` and `CRL` in `Labs`; `SONORA_QUEST` in `AppointmentProvider` and `ClientFacingLabs`; `SAMSUNG_HEALTH` in `ManualProviders`; `TANDEM_SOURCE` in `PasswordProviders`; `APERO` and `PVERIFY` in `PayorCodeExternalProvider`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; and several corrected/blocked status values in `OrderStatus`.
* The SDK now includes a `CompendiumClient` (and async variant) with `search` and `convert` methods for querying the v3 compendium API. Request builder classes across activity, body, and aggregate resources now support `additionalProperty(key, value)` and `additionalProperties(map)` for passing custom key-value pairs. A new `SseEventParser` utility is also available for advanced SSE parsing with discriminated union support.
* The SDK now includes a new `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving lab account information. Two new request types — `SearchCompendiumBody` and `ConvertCompendiumBody` — have been added to support compendium search and lab conversion operations. Several existing request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing custom query parameters.
* The SDK now includes a new `RawLabAccountClient` for retrieving team lab accounts, with support for optional `lab_account_id` and `status` filters. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* Request builder classes across the SDK now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to requests. This enables forward-compatibility with new API parameters before they are formally modeled in the SDK.
* Request builder classes across the `user` and `vitals` resources now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra fields to requests, providing forward compatibility with new API parameters not yet modeled in the SDK.
* The builder for all vitals request classes (e.g., `VitalsHeartrateRequest`, `VitalsGlucoseRequest`, `VitalsCholesterolLdlRequest`, and others) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface. These new optional builder methods allow consumers to attach arbitrary key-value metadata to requests without affecting existing usage.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on both the `_FinalStage` interface and `Builder` implementation. This allows SDK consumers to attach arbitrary extra parameters to any Vitals request without waiting for explicit SDK support.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on all request and model classes, making it straightforward to attach custom fields without subclassing. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type representing a canonical lab test candidate with LOINC, CPT, and provider metadata. `ClientFacingAppointment` has a new optional `appointmentNotes` field. Builder classes across many response and event types now expose `additionalProperty` and `additionalProperties` methods for setting arbitrary extra fields.
* Builder interfaces and implementations for `ClientFacing*` model types (including `ClientFacingBody`, `ClientFacingBloodPressureTimeseries`, `ClientFacingAppointmentEvent`, `ClientFacingAtHomePhlebotomyOrder`, and many others) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new optional builder methods allow consumers to attach arbitrary metadata when constructing these objects programmatically.
* Builder objects for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easy to attach arbitrary metadata when constructing SDK model objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* The builder for numerous `ClientFacing*` types (ECG, Fall, Fat, FloorsClimbed, Food, ForcedExpiratoryVolume1, ForcedVitalCapacity, Glucose, Handwashing, HeartRate, and related event/timeseries variants) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach custom metadata to objects. A new `INSULIN_PUMP` source type has also been added to `ClientFacingElectrocardiogramSourceType`.
* Builder interfaces for all `ClientFacing*` types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields when building objects. Two new enums have been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (values: `MEAL`, `CORRECTION`, `MIXED`, `UNKNOWN`) and `ClientFacingInsulinInjectionSampleDeliveryMode` (values: `BASAL`, `BOLUS`).
* The SDK now includes three new types: `ClientFacingLabAccount` (for lab account management), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (for handling lab report parsing job webhook events). Additionally, the builder `_FinalStage` interfaces for several existing types (`ClientFacingLabLocation`, `ClientFacingLeanBodyMassChanged`, `ClientFacingLeanBodyMassHistoricalPullCompleted`, `ClientFacingLeanBodyMassSample`, `ClientFacingLoinc`, `ClientFacingMarker`, `ClientFacingMarkerComplete`, and `ClientFacingMealChanged`) now expose `additionalProperty` and `additionalProperties` methods for more flexible object construction.
* Builder objects for numerous `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary metadata when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingMenstrualCycleSourceType` enum.
* The `ClientFacingSleep` type now includes a new optional `recoveryReadinessScore` field (0–100), representing the provider's recovery/readiness proxy sourced from Oura, Whoop, and Ultrahuman. Builder classes across many types now expose `additionalProperty` and `additionalProperties` methods for attaching arbitrary metadata. A new `INSULIN_PUMP` value has been added to the `ClientFacingSleepCycleSourceType` enum.
* Builders for all `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary metadata to objects at build time. These methods are also declared on the corresponding `_FinalStage` interfaces, making them available throughout the staged builder API.
* Builder classes for numerous `ClientFacing*` and response types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields when constructing objects via the builder pattern.

Additionally, builder interfaces across many existing model classes now expose `additionalProperty` and `additionalProperties` methods, allowing consumers to attach arbitrary extra properties when constructing request objects.
