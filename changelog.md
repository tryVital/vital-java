## 2.0.0 - 2026-04-02
* The `parserCreateJob(InputStream, String, ...)` overloads (including variants accepting `MediaType` and `RequestOptions`) have been removed from `LabReportClient` and `AsyncLabReportClient`. Replace these calls with the `parserCreateJob(File, BodyCreateLabReportParserJob)` overload instead. Additionally, the `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum — update any switch statements or references accordingly. New opt-in SDK request/response logging is now available by passing a `LogConfig` to `ClientOptions.Builder.logging()`.
* Several breaking changes have been made to the lab-related request types:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` parameter type has changed from `String` to `AppointmentPscLabs`. Callers must replace raw string values with the corresponding `AppointmentPscLabs` enum constant.
* **`BulkExportConnectionsBody`**: The `teamId` field and its builder methods (`teamId(...)`) have been removed. Remove any calls to `.teamId(...)` on this builder.
* **`RawLabReportClient`**: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Migrate to the `File`-based `parserCreateJob(File, ...)` overloads instead.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* The `getTeamId()` method and corresponding `teamId()` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. The `getAddress()` method on `UserInfoCreateRequest` now returns `UserAddress` instead of `Address` — callers must update their type references accordingly. New optional fields `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions` have been added to `ManualConnectionData`, and an optional `accessNotes` field has been added to `Address`.
* The `AggregateExprArg.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface will fail to compile until these methods are added.
* Additionally, `LinkBulkPauseRequestTeamId` (in `com.vital.api.resources.link.types`) has been removed and replaced by `AppointmentPscLabs` (in `com.vital.api.types`), and `LinkBulkTriggerHistoricalPullRequestTeamId` has been removed and replaced by `ClientFacingInsulinInjectionSampleDeliveryForm`.
* New optional fields `deliveryMode`, `deliveryForm`, and `bolusPurpose` are now available on `ClientFacingInsulinInjectionSample`, and `appointmentNotes` is now available on `AppointmentRescheduleRequest`.
* **Breaking changes in this release:**
* `ParsingJob.getJobId()` has been removed along with the `JobIdStage` builder interface. Callers that set `jobId` via the builder must remove that step — `id()` now advances directly to `StatusStage`.
* The `Placeholder` builder no longer has a required `placeholder(boolean)` stage; `Placeholder.builder()` now returns a `Builder` directly (the `PlaceholderStage` interface has been removed).
* The `QueryGroupByItem.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any exhaustive `Visitor` implementation must add these methods.
**New capabilities:**
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now expose an optional `accessNotes` field (`getAccessNotes()` / `accessNotes()` builder method).
* `QueryGroupByItem` supports three new expression types: `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`.
* Several breaking changes have been made to core SDK types:
* **`QuerySelectItem.Visitor`** now requires three new `visit()` methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing this interface must be updated to add these methods.
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Update any code that calls this method or passes an `Address` to the builder's `address()` stage.
* **`UserRefreshSuccessResponse` and `UserRefreshErrorResponse` builders** no longer have a `SuccessStage` or `success(boolean)` method. Remove calls to `.success(...)` in builder chains — `builder()` now returns `UserIdStage` directly and the `success` field is now a computed constant.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** have been removed; use the new `ResultMetadataGender` and `SearchMode` enums instead.
* New optional `accessNotes` field added to `UsAddress`.
* The `getMarkers` method now supports a new optional `lab_slug` filter parameter, allowing callers to scope marker results to a specific lab by slug. Javadoc on `getMarkers` and `getMarkersForLabTest` overloads has also been updated to accurately describe endpoint behavior, including that `getMarkers` excludes reflex markers and `getMarkersForLabTest` includes them.
* The `getMarkers` method now supports filtering by `lab_slug` via the `LabTestsGetMarkersRequest` builder. Additionally, the `getMarkersForLabTest` methods now include Javadoc clarifying that reflex markers are included in results, while `getMarkers` notes that reflex markers are excluded.
* The link provider/manual endpoint now supports two new optional request headers: `x-vital-ios-sdk-version` and `x-vital-android-sdk-version`, allowing callers to forward their mobile SDK version to the Vital API. The `team_id` query parameter has been removed from several bulk link endpoints (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`).
* The SDK now supports configurable logging via the new `LogConfig` class. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request and response logging with adjustable log levels and custom logger implementations (silent by default).
* Two new resource clients are now available: `CompendiumClient` (accessed via `vital.compendium()`) and `LabAccountClient` (accessed via `vital.labAccount()`).
* Several enums have been expanded with new values: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; multiple new `CORRECTED` and `LAB_PROCESSING_BLOCKED` statuses in `OrderStatus`; `TANDEM_SOURCE` in `PasswordProviders`; and `APERO` and `PVERIFY` in `PayorCodeExternalProvider`.
* The SDK now includes a `CompendiumClient` (and async counterpart `AsyncCompendiumClient`) for searching and converting lab test compendium entries via `search()` and `convert()` methods. Request builder classes for activity, aggregate, and body resources now expose `additionalProperty()` and `additionalProperties()` builder methods for attaching custom key/value pairs to requests.
* New `LabAccountClient` (and async variant) added to the SDK, exposing a `getTeamLabAccounts()` method for retrieving team lab account information. Two new compendium request types — `SearchCompendiumBody` and `ConvertCompendiumBody` — are now available for compendium search and conversion operations. Several existing request builder classes also gain `additionalProperty` and `additionalProperties` helpers for passing through arbitrary extra fields.
* The SDK now includes a new `RawLabAccountClient` for direct HTTP access to the lab accounts API via `getTeamLabAccounts()`, returning the raw HTTP response. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builders now expose `additionalProperty` and `additionalProperties` methods for passing extra parameters.
* Request builder types across the SDK now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to API requests. This is available on all request and payload classes in the `labtests`, `link`, `meal`, `menstrualcycle`, `order`, `payor`, `profile`, `providers`, `sleep`, `sleepcycle`, `team`, `testkit`, and `user` resources.
* Request builder classes across the `user` and `vitals` resources now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra properties to requests, providing forward compatibility with API fields not yet modeled in the SDK.
* The builder for all `Vitals*Request` classes now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods via the `_FinalStage` interface. Consumers can use these methods to attach arbitrary extra properties to request objects when building them.
* Vitals request builders now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. This allows SDK consumers to attach arbitrary key-value properties to vitals requests across all supported vitals types.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on the `_FinalStage` interface and `Builder` class for all major request and type objects, enabling consumers to attach arbitrary extra properties during construction. A new `AwakeningsValueMacroExpr` type is now available, and `AppointmentBookingRequest` supports a new optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type representing LOINC-based canonical lab test candidates, complete with fields for LOINC set hash, display name, aliases, codes, and confidence scores. `ClientFacingAppointment` gains a new optional `appointmentNotes` field. Builder classes across many response types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods for setting extra properties programmatically.
* Builder classes for `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary metadata when constructing these objects via the builder pattern. These methods are also declared on the `_FinalStage` interface. All existing code continues to compile and run without changes.
* Builder instances for many `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, making it easier to attach arbitrary metadata when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* Builder objects for many `ClientFacing` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary extra fields when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* Builder instances for `ClientFacing*` types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields when constructing these objects. Two new enums have also been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (MEAL, CORRECTION, MIXED, UNKNOWN) and `ClientFacingInsulinInjectionSampleDeliveryMode` (BASAL, BOLUS).
* Three new public types have been added: `ClientFacingLabAccount` (representing a lab account with billing, delegated flow, and allowlist fields), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (webhook event types for lab report parsing jobs). Additionally, `additionalProperty` and `additionalProperties` builder methods are now available on the `_FinalStage` interface of several existing types including `ClientFacingLabLocation`, `ClientFacingLeanBodyMassSample`, `ClientFacingMarker`, `ClientFacingMarkerComplete`, and others.

A new `INSULIN_PUMP` value has also been added to the `ClientFacingMenstrualCycleSourceType` enum.
* The `ClientFacingSleep` type now includes a new optional `recoveryReadinessScore` field (0–100), representing a provider's recovery/readiness proxy sourced from Oura, Whoop, and Ultrahuman. A new `INSULIN_PUMP` value is available on the `ClientFacingSleepCycleSourceType` enum. Additionally, `additionalProperty` and `additionalProperties` builder methods are now exposed on the `_FinalStage` interface of many `ClientFacing*` types, allowing consumers to attach arbitrary extra properties when constructing these objects.

These allow SDK consumers to attach arbitrary extra fields when constructing objects, improving forward-compatibility with API fields not yet modeled in the SDK.
