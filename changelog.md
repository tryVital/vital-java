## 2.0.0 - 2026-04-01
* The `InputStream`-based `parserCreateJob` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers should migrate to the equivalent `File`-based overloads (`parserCreateJob(File file, BodyCreateLabReportParserJob request)`). Additionally, the `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum — any code referencing this constant or using it in a switch statement must be updated. New in this release: the SDK now supports optional structured logging via `ClientOptions.Builder.logging(LogConfig)`, and `Stream` adds support for SSE event-discriminated union payloads via `Stream.fromSseWithEventDiscrimination(...)`.
* Several breaking changes have been made to request types in the lab tests and link resources:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field type has changed from `String` to the new `AppointmentPscLabs` enum. Update all calls to `.lab(...)` to pass an `AppointmentPscLabs` value instead of a plain string.
* **`BulkExportConnectionsBody`**: The `teamId` field has been removed. Remove any calls to `.teamId(...)` in builder usage and the `getTeamId()` accessor.
* **`RawLabReportClient`**: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Migrate to the `File`-based `parserCreateJob(File, ...)` overloads instead.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* The `teamId` field and its builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Additionally, `UserInfoCreateRequest.getAddress()` now returns `UserAddress` instead of `Address` — callers must update their type declarations accordingly. To migrate, remove any `.teamId(...)` builder calls from bulk link operation request objects and update variable types referencing the `address` field of `UserInfoCreateRequest` to use `UserAddress`.
* The `AggregateExprArg.Visitor` interface has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing `Visitor` must be updated to add these methods.
* In addition, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields (with new enum types `ClientFacingInsulinInjectionSampleDeliveryMode`, `ClientFacingInsulinInjectionSampleDeliveryForm`, and `ClientFacingInsulinInjectionSampleBolusPurpose`), `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field, and a new `AppointmentPscLabs` enum (`QUEST`, `SONORA_QUEST`) is introduced.
* Several breaking changes have been made to order and lab result types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(...)`, `getParentId()`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added to the staged builder — callers must now set `lastEvent(...)` after `updatedAt(...)`. A new optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The `status` field and builder method have changed from `String` to the new `OrderTransactionStatus` enum type.
* **`HistoricalPullCompleted`**: The `isFinal` builder stage (`IsFinalStage`) and `isFinal(boolean)` setter have been removed; `getIsFinal()` now always returns `true`.
* New additions include `statusDetail` on `ClientFacingOrderEvent`, `orderTransaction` on `LabResultsRaw`, and the new `LabAccountStatus` enum.
* Several breaking changes are included in this release:
* **`ParsingJob`**: The `job_id` field, `getJobId()` getter, and the `JobIdStage` builder interface have been removed. Update builder chains that called `.id(...).jobId(...)` to call `.id(...).status(...)` directly, and remove any reads of `getJobId()`.
* **`Placeholder`**: The `PlaceholderStage` interface and the `placeholder(boolean)` builder setter no longer exist. `getPlaceholder()` now always returns `true`; update any builder usage accordingly.
* **`QueryGroupByItem.Visitor`**: Three new visit methods have been added — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. All existing `Visitor` implementations must implement these new methods.
* New optional `accessNotes` field is now available on `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`, and all affected builders now expose `additionalProperty`/`additionalProperties` helpers.
* Several breaking changes have been made to the SDK's type model:
* **`UserInfo.address`** now has type `UserAddress` instead of `Address`; callers must update usages of `AddressStage.address()` to pass a `UserAddress` instance.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse` builders**: the `SuccessStage` interface and `success(boolean)` builder method have been removed. Call `builder()` directly (it now returns `UserIdStage`) and remove any calls to `.success(...)` in builder chains. `getSuccess()` now returns a hardcoded `Boolean` constant (`false` and `true` respectively).
* **`QuerySelectItem.Visitor`** has three new required methods — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)` — that all `Visitor` implementations must add.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** (in `com.vital.api.resources.link.types`) have been replaced by `ResultMetadataGender` and `SearchMode` in `com.vital.api.types`; update imports accordingly.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter, allowing consumers to scope marker results to a specific lab by its slug. Javadoc descriptions for `getMarkers` and `getMarkersForLabTest` have also been updated to clarify behavior around reflex markers.
* The `getMarkers` endpoint now accepts an optional `lab_slug` parameter, allowing consumers to filter markers by lab slug. Documentation for `getMarkersForLabTest` has been added to clarify that reflex markers are included in its results.
* The `connectManualProvider` method on `LinkClient` now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, allowing mobile SDK version information to be forwarded to the Vital API. Additionally, the `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`).
* The SDK now supports configurable logging via the new `LogConfig` class — pass it to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response tracing (silent by default). Two new resource clients are available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async equivalents. New enum values have been added across `AppointmentProvider`, `ClientFacingLabs`, `Labs`, `ManualProviders`, `PasswordProviders`, `PayorCodeExternalProvider`, `OrderLowLevelStatus`, and `OrderStatus` to reflect expanded provider and order lifecycle support.
* The SDK now includes a `CompendiumClient` (and async counterpart) with `search()` and `convert()` methods for interacting with the `/v3/compendium` endpoints. A new `SseEventParser` utility class is also available for parsing Server-Sent Events with support for both data-level and event-level discriminated unions. Request builder types across Activity, Body, Aggregate, and QueryBatch now expose `additionalProperty` and `additionalProperties` methods for passing arbitrary extra fields.
* The SDK now includes a `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request body types — `SearchCompendiumBody` and `ConvertCompendiumBody` — are available for structured compendium search and lab conversion operations. Several existing request builders also gain `additionalProperty` / `additionalProperties` helpers for supplying arbitrary parameters.
* The SDK now includes a new `RawLabAccountClient` with a `getTeamLabAccounts()` method for retrieving team lab accounts, filterable by `lab_account_id` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. Additionally, `additionalProperty` and `additionalProperties` builder methods are now available on all affected lab test request builder classes for passing through custom properties.
* Request builder classes across multiple resources now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to request objects without requiring SDK changes. This affects request types in the `labtests`, `link`, `meal`, `menstrualcycle`, `order`, `payor`, `profile`, `providers`, `sleep`, `sleepcycle`, `team`, `testkit`, and `user` packages.
* Builder classes for all user and vitals request types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, enabling consumers to attach arbitrary key-value pairs to requests for greater forward compatibility.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface, allowing consumers to attach arbitrary extra properties when constructing requests. This update applies uniformly across all Vitals request types, including cholesterol, glucose, heart rate, HRV, hypnogram, distance, floors climbed, and more.
* Vitals request builders now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface, enabling consumers to attach arbitrary extra properties to any vitals request when using the fluent builder pattern.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` builder methods on all request and model builder classes, making it easy to attach arbitrary extra fields. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for lab test canonical matching, supporting LOINC set hashes, display names, aliases, CPT codes, and confidence scoring. An optional `appointmentNotes` field has been added to `ClientFacingAppointment`. Builder classes across many response and event types now expose `additionalProperty` and `additionalProperties` methods for setting arbitrary extra fields programmatically.
* Builder instances for `ClientFacing*` model types (such as `ClientFacingAppointmentEvent`, `ClientFacingBody`, `ClientFacingBloodPressureTimeseries`, and many others) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary extra properties when constructing these objects.

The `ClientFacingDeviceSourceType` enum has also been extended with a new `INSULIN_PUMP` value.

The `ClientFacingElectrocardiogramSourceType` enum gains a new `INSULIN_PUMP` value.
