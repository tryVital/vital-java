## 2.0.0 - 2026-04-02
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, String)`, `parserCreateJob(InputStream, String, MediaType)`, `parserCreateJob(InputStream, String, RequestOptions)`, and `parserCreateJob(InputStream, String, MediaType, RequestOptions)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Migrate to the `parserCreateJob(File, BodyCreateLabReportParserJob)` overload instead.
* The `INSULIN_INJECTION` enum constant has been removed from `IntervalTimeseriesExprTimeseries`. Update any switch statements or references to this value accordingly.
* New in this release: the SDK now supports configurable request/response logging via a new `logging(LogConfig)` option on the client builder (silent by default). SSE streams now support event-level discrimination via the new `Stream.fromSseWithEventDiscrimination()` factory methods.
* Several breaking changes have been made to lab test and link request types:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field type has changed from `String` to the new `AppointmentPscLabs` enum. Update calls to `.lab(...)` to pass an `AppointmentPscLabs` value instead of a plain string.
* **`BulkExportConnectionsBody`**: The `teamId` field and its builder methods (`getTeamId()`, `teamId(...)`) have been removed. Remove any references to these from your code.
* **`RawLabReportClient`** (raw client): The `parserCreateJob(InputStream, ...)` overloads have been removed. Use the `File`-based `parserCreateJob` overloads instead.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* **`UserInfoCreateRequest`**: The `address` field type has changed from `Address` to `UserAddress`. Callers must update any code that passes or receives an `Address` instance for this field to use `UserAddress` instead.
* **`BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, `LinkListBulkOpsRequest`**: The `teamId` field and its associated `getTeamId()` getter and builder methods have been removed. Any code setting or reading `teamId` on these classes must be updated.
* New optional fields have also been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* Three new variants — `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr` — have been added to the `AggregateExprArg.Visitor` interface. Any existing implementation of this interface must add handler methods for all three new variants to continue compiling. Additionally, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields, `AppointmentRescheduleRequest` supports an optional `appointmentNotes` field, and two new enums (`AppointmentPscLabs`, `ClientFacingInsulinInjectionSampleDeliveryForm`) have been added.
* Several breaking changes have been made to order and lab result types:
* **`ClientFacingOrder`**: The `getParentId()` method and its builder methods (`parentId(...)`) have been removed. Additionally, the builder now requires a new mandatory `lastEvent(ClientFacingOrderEvent)` step inserted between `updatedAt(...)` and `hasAbn(...)` — existing builder call chains will fail to compile without adding this step. Two new fields are added: `clinicalNotes` (optional) and `lastEvent` (required, replacing `status` for event tracking).
* **`GetOrderTransactionResponse`**: `getStatus()` now returns the typed enum `OrderTransactionStatus` instead of `String`. Update callers to use the enum type, and update builder calls to pass `OrderTransactionStatus` instead of a raw string.
* **`HistoricalPullCompleted`**: The `isFinal` field has been removed from the model and builder. The `getIsFinal()` method now always returns `true` as a `Boolean`. Remove any calls to `isFinal(...)` on the builder.
* **`LinkBulkExportRequestTeamId`** enum has been removed from `com.vital.api.resources.link.types`.
* **New**: `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been added. `ClientFacingOrderEvent` gains an optional `statusDetail` field. `LabResultsRaw` gains an optional `orderTransaction` field.
* **Breaking changes:**
* `ParsingJob.getJobId()` has been removed along with the `JobIdStage` builder interface. Builder chains that previously called `.id(...).jobId(...).status(...)` must now call `.id(...).status(...)` directly.
* `Placeholder`'s required `placeholder` field and `PlaceholderStage` builder interface have been removed. `getPlaceholder()` now always returns `true` and no argument is needed when building a `Placeholder`.
* Three new `visit(...)` methods have been added to `QueryGroupByItem.Visitor` (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). Any exhaustive `Visitor` implementation must be updated to implement these new methods.
**New features:**
* `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now expose an optional `accessNotes` field via `getAccessNotes()` and builder methods.
* All affected builder types now support `additionalProperty(String, Object)` and `additionalProperties(Map)` helper methods on their `_FinalStage`.
* Several breaking changes have been made to existing types.
* **`UserInfo.address`** now has type `UserAddress` instead of `Address`; any code that calls `getAddress()` or passes an `Address` to the builder's `address()` stage must be updated to use `UserAddress`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** builder entry point has changed: the required `SuccessStage` (`.success(boolean)`) has been removed. Call `builder()` directly and proceed from `UserIdStage`. The `getSuccess()` method now returns a constant `Boolean` instead of a primitive `boolean`.
* **`QuerySelectItem.Visitor`** interface has three new required methods (`visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, `visit(InsulinInjectionTimeseriesExpr)`); any existing `Visitor` implementations must implement these new methods.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** enums have been removed; they are replaced by the new `ResultMetadataGender` and `SearchMode` enums in `com.vital.api.types`.
* New additions include the `ResultMetadataGender` enum, the `SearchMode` enum, and an optional `accessNotes` field on `UsAddress`.
* The `getMarkers` method now supports filtering by `lab_slug` via `LabTestsGetMarkersRequest`. Javadoc descriptions for `getMarkers` and `getMarkersForLabTest` have been updated to accurately reflect their behavior, including noting that `getMarkers` excludes reflex markers and that `getMarkersForLabTest` includes them.
* The `getMarkers` method now supports a new optional `lab_slug` filter parameter, allowing consumers to filter markers by lab slug. The `getMarkersForLabTest` methods now include Javadoc clarifying that reflex markers are included in the response. Additionally, a bug fix ensures the `drawCompleted` PATCH request correctly sends an empty request body.
* The link provider manual endpoint now supports two new optional request headers — `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` — allowing mobile SDK clients to identify their platform version when connecting providers. Additionally, the `team_id` query parameter has been removed from several bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`).
* The SDK now supports configurable logging via a new `LogConfig` API — pass a `LogConfig` instance to `VitalBuilder.logging()` to enable request/response logging with customizable log levels and logger implementations (silent by default). Two new resource clients are available: `CompendiumClient` (via `vital.compendium()`) and `LabAccountClient` (via `vital.labAccount()`). Several enums have been expanded with new values including `SONORA_QUEST` (AppointmentProvider, ClientFacingLabs, Labs), `CRL` (Labs), `SAMSUNG_HEALTH` (ManualProviders), `TANDEM_SOURCE` (PasswordProviders), `APERO`/`PVERIFY` (PayorCodeExternalProvider), `CORRECTED`/`LAB_PROCESSING_BLOCKED` (OrderLowLevelStatus), and multiple new `CORRECTED` and `LAB_PROCESSING_BLOCKED` statuses in `OrderStatus`.
* The SDK now includes a new `CompendiumClient` (and async variant) with `search()` and `convert()` methods for interacting with the `/v3/compendium` endpoints. A new `SseEventParser` utility has also been added to support Server-Sent Events parsing with discriminated union types. Additionally, request builder interfaces now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* The SDK now includes a new `LabAccountClient` (and async variant) exposing a `getTeamLabAccounts()` method for retrieving team lab account data. Two new request types, `SearchCompendiumBody` and `ConvertCompendiumBody`, have been added for compendium search and conversion operations. Several existing request builders also gain `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* The SDK now includes a new `RawLabAccountClient` for direct HTTP access to the lab account API (`GET /v3/lab_test/lab_account`). An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test and lab report request builders now expose `additionalProperty` and `additionalProperties` methods for passing through arbitrary extra fields.
* All request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new optional builder methods allow SDK consumers to attach arbitrary extra fields to requests, enabling forward compatibility with API fields not yet formally modeled in the SDK.
* Builder classes for user and vitals request objects now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to requests. This makes it easier to pass new or undocumented API parameters without requiring an SDK update.
* All Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementation. This allows consumers to attach arbitrary extra properties to requests without changes to the typed API surface.
* Vitals request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder`. This allows consumers to attach arbitrary key-value pairs to requests when constructing them via the fluent builder API.

A new `AwakeningsValueMacroExpr` class has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field via `getAppointmentNotes()` / `appointmentNotes(String)`.
* The SDK now includes a new `CanonicalCandidate` type for lab test canonicalization, carrying LOINC/CPT codes, aliases, popularity score, and confidence fields. `ClientFacingAppointment` has a new optional `appointmentNotes` field. All builder classes now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` convenience methods for setting arbitrary extra properties.
