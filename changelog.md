## 2.0.0 - 2026-04-01
* **Breaking changes:**
* The `parserCreateJob(InputStream, String)`, `parserCreateJob(InputStream, String, MediaType)`, `parserCreateJob(InputStream, String, RequestOptions)`, and `parserCreateJob(InputStream, String, MediaType, RequestOptions)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Replace these calls with the `File`-based `parserCreateJob(File, BodyCreateLabReportParserJob)` overloads instead.
* `IntervalTimeseriesExprTimeseries.INSULIN_INJECTION` has been removed from the enum. Update any references or switch statements that use this value.
**New features:**
* SDK logging is now configurable via `ClientOptions.Builder.logging(LogConfig)`. By default the SDK remains silent; pass a `LogConfig` to enable request/response logging.
* `Stream` now supports SSE event-level discrimination via the new `Stream.fromSseWithEventDiscrimination(...)` factory methods, enabling deserialization of discriminated-union SSE payloads.
* Several breaking changes have been introduced in this release:
* The `InputStream`-based overloads of `parserCreateJob` on `RawLabReportClient` have been removed. Callers must migrate to the `File`-based overloads.
* The `lab` parameter in `LabTestsGetPscAppointmentAvailabilityRequest` has changed from `String` to the new `AppointmentPscLabs` enum type; existing string values must be replaced with the corresponding enum constant.
* The `teamId` field (of type `LinkBulkExportRequestTeamId`) has been removed from `BulkExportConnectionsBody`; callers setting this field must remove those calls.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes in this release.
* The `teamId` field (and its getter `getTeamId()` and builder methods) has been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Remove any `.teamId(...)` builder calls and `getTeamId()` usages from your code.
* The `address` field of `UserInfoCreateRequest` has changed its type from `Address` to `UserAddress`. Update your code to pass a `UserAddress` instance and update any references to the `getAddress()` return type accordingly.
* New optional fields have also been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* The `AggregateExprArg.Visitor<T>` interface now requires three additional `visit` methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface must add these three methods to compile successfully. Additionally, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields, and `AppointmentRescheduleRequest` supports an optional `appointmentNotes` field.
* Several breaking changes have been made to order-related model types:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)`, `parentId(Optional<String>)`) have been removed. Two new fields have been added: `clinicalNotes` (optional `String`) and `lastEvent` (required `ClientFacingOrderEvent`). The builder stage chain has changed — `updatedAt()` now returns a `LastEventStage` (requiring `.lastEvent(...)`) before proceeding to `HasAbnStage`.
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` instead of `String`. Builder code passing a raw `String` to `status()` must be updated to pass an `OrderTransactionStatus` value.
* **`HistoricalPullCompleted`**: The `IsFinalStage` builder stage and `isFinal(boolean)` method have been removed. The `is_final` field now always returns `true`. Remove `.isFinal(...)` calls from any builder chains.
* New additions include an optional `statusDetail` field on `ClientFacingOrderEvent`, an optional `orderTransaction` field on `LabResultsRaw`, and a new `LabAccountStatus` enum.
* Several breaking changes are included in this release:
* **`ParsingJob`**: The `jobId` field and `getJobId()` method have been removed. The staged builder no longer includes a `JobIdStage` step — remove `.jobId(...)` calls from any `ParsingJob.builder()` chains.
* **`Placeholder`**: The `placeholder` field and required `PlaceholderStage` builder interface have been removed; the builder is now a simple flat `Builder` class.
* **`QueryGroupByItem.Visitor`**: Three new required visitor methods have been added — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)` — which must be implemented by any class implementing the `Visitor` interface.
* New optional `accessNotes` field is now available on `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* Several breaking changes have been introduced in this release:
* **`QuerySelectItem.Visitor`** has three new required methods (`visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, `visit(InsulinInjectionTimeseriesExpr)`). Any class implementing `Visitor` must add these methods or it will fail to compile.
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Code that assigns or passes the result as `Address` must be updated to use `UserAddress`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** no longer expose a `success` field via their builders. The `SuccessStage` interface and `success(boolean)` builder method have been removed; callers must remove references to these. `getSuccess()` now returns a hardcoded `Boolean` value.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** (in `com.vital.api.resources.link.types`) have been removed; replace with the new `ResultMetadataGender` and `SearchMode` enums in `com.vital.api.types`.
* New additions include the `accessNotes` optional field on `UsAddress`, three new `QuerySelectItem` expression types, and the `ResultMetadataGender` and `SearchMode` enums.
* The `getMarkers` method now supports an optional `lab_slug` query parameter for filtering markers by lab slug. Javadoc for `getMarkers` and `getMarkersForLabTest` has been updated to clarify that `getMarkers` excludes reflex markers while `getMarkersForLabTest` includes them.
* The `getMarkers` method on `LabTestsClient` now supports an optional `lab_slug` query parameter, allowing consumers to filter markers by lab slug. Documentation for `getMarkersForLabTest` has been updated to clarify that reflex markers are included in the results.
* The `LinkClient` now supports sending optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` headers when calling the link provider manual endpoint. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, and `bulk_pause`).
* The SDK now supports configurable logging via a new `LogConfig` API — pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response logging with customizable log level and logger implementation (silent by default). Two new resource clients are available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async equivalents. Several enums have been expanded with new values: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `TANDEM_SOURCE` in `PasswordProviders`; `APERO` and `PVERIFY` in `PayorCodeExternalProvider`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; and five new `CORRECTED`/`LAB_PROCESSING_BLOCKED` statuses in `OrderStatus`.
* The SDK now includes a `CompendiumClient` (and async variant) with `search` and `convert` methods for interacting with the Vital Compendium API. Request builder types across activity, body, and aggregate resources now expose `additionalProperty` and `additionalProperties` methods, making it easier to pass custom fields when constructing requests. A new `SseEventParser` utility class has also been added to support Server-Sent Events with discriminated union deserialization.
