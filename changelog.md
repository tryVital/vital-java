## 2.0.0 - 2026-03-31
* The SDK now supports optional logging configuration through the new `LogConfig` parameter in `ClientOptions.Builder`. Logging is silent by default but can be enabled for debugging HTTP requests and responses. Enhanced streaming capabilities with new SSE event-level discrimination support for handling discriminated unions in server-sent events. Removed deprecated `InputStream`-based file upload methods from `LabReportClient` - use `File`-based methods instead.
* The `parserCreateJob()` method overloads that accepted `InputStream` parameters have been removed from `RawLabReportClient`. Use the `File` parameter overloads instead. Additionally, new optional fields `labAccountId` and `labSlug` are now available on lab test creation requests for enhanced lab specification.
* The `team_id` parameter has been removed from bulk operations (`BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`) and list requests (`LinkListBulkOpsRequest`). Remove any usage of `teamId()` builder methods when creating these request objects. New optional fields are available: `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions` on `ManualConnectionData`, and `accessNotes` on the `Address` type.
* The SDK now supports enhanced insulin injection tracking with delivery metadata (delivery mode, delivery form, and bolus purpose fields). New aggregate expression types are available for derived readiness columns and insulin injection timeseries queries. Appointment rescheduling requests now support optional appointment notes, and new PSC Labs provider options (Quest and Sonora Quest) are available.
* Breaking changes:
* The `ClientFacingOrder` constructor now requires a `lastEvent` parameter. The existing `status` field is deprecated - use `lastEvent` instead for accessing the latest order event.
* The `parentId` field has been removed from `ClientFacingOrder`.
* The `jobId` field has been removed from `ParsingJob`.
* `GetOrderTransactionResponse.getStatus()` now returns `OrderTransactionStatus` enum instead of `String`.
* New features:
* Added `clinicalNotes` field to `ClientFacingOrder` for storing clinical notes.
* Added `statusDetail` field to `ClientFacingOrderEvent` for additional status information.
* Added `orderTransaction` field to `LabResultsRaw`.
* Added new `LabAccountStatus` enum with values: `ACTIVE`, `PENDING`, `SUSPENDED`.
* The SDK now supports an optional `accessNotes` field on patient address types to provide delivery instructions or access information. Query expression union types have been extended with new variants for insulin injection timeseries, awakening value macros, and derived readiness columns.
* New `accessNotes` field available on `UsAddress` for additional delivery instructions. The `UserInfo` class now uses the more specific `UserAddress` type instead of the generic `Address` type for improved type safety.
* The lab tests client now supports filtering markers by lab slug using the new `lab_slug` parameter in the `getMarkers()` method. Updated documentation provides clearer descriptions for marker-related endpoints.
* The SDK now supports sending iOS and Android SDK version information through optional headers for better tracking and debugging support.
* The SDK now supports comprehensive logging with configurable levels (DEBUG, INFO, WARN, ERROR) and custom logger implementations. Logging is silent by default and can be enabled via `VitalBuilder.logging()`. New client methods `compendium()` and `labAccount()` are now available. Additional laboratory providers (SONORA_QUEST, CRL), manual providers (SAMSUNG_HEALTH), order statuses (CORRECTED, LAB_PROCESSING_BLOCKED), and payor code providers (APERO, PVERIFY) have been added to support expanded service integrations.
* The SDK now includes a new Compendium API client with search() and convert() methods for exercise and nutrition data lookups. Additionally, comprehensive Server-Sent Events parsing utilities have been added to support real-time data streaming with discriminated union handling.
* New compendium search and lab account management capabilities. The SDK now supports searching for lab tests using various criteria (text queries, CPT codes, LOINC codes) and converting tests between different lab providers. A new LabAccountClient is available for managing team lab accounts.
* The SDK now includes a new `RawLabAccountClient` with `getTeamLabAccounts()` method for retrieving team lab accounts. Additionally, lab order creation now supports an optional `clinicalNotes` field in `CreateOrderRequestCompatible`.
* New `additionalProperty()` and `additionalProperties()` methods available on all request builder classes, enabling custom field extensions for API requests.
* New builder methods `additionalProperty()` and `additionalProperties()` are now available on all request builders, enabling dynamic extension of request objects with custom key-value pairs.
* The SDK now supports additional properties on all vitals request builders. Use `additionalProperty()` to add individual key-value pairs or `additionalProperties()` to add multiple properties at once for extended request customization.
* New `additionalProperty()` and `additionalProperties()` builder methods are now available on all vitals request classes. These methods allow you to pass custom key-value pairs for extensibility and metadata.
* New builder methods are now available for setting additional properties on request and response objects. Use `additionalProperty(key, value)` to set individual properties or `additionalProperties(map)` to set multiple properties at once. A new `appointmentNotes` field has been added to appointment booking requests.
* New CanonicalCandidate type available for lab test standardization with LOINC code mapping. The ClientFacingAppointment type now includes an optional appointmentNotes field. All builder classes now support additionalProperty() and additionalProperties() methods for flexible metadata handling.
* The SDK now supports setting additional properties on all ClientFacing objects through new builder methods. Use `additionalProperty(key, value)` to set individual properties or `additionalProperties(map)` to set multiple properties at once.
* The SDK now supports setting custom additional properties on model objects through new `additionalProperty()` and `additionalProperties()` builder methods. A new `INSULIN_PUMP` device source type is also available.
* The SDK now supports setting additional properties on client-facing types using new builder methods `additionalProperty()` and `additionalProperties()`. The `ClientFacingElectrocardiogramSourceType` enum now includes `INSULIN_PUMP` as a valid source type option.
* The SDK now supports setting additional properties on builder objects through new `additionalProperty()` and `additionalProperties()` methods. This enables extending objects with custom key-value pairs while maintaining type safety.
* New lab report parsing job webhook events are now available (`ClientFacingLabReportParsingJobCreatedEvent` and `ClientFacingLabReportParsingJobUpdatedEvent`) for tracking lab report processing status. The SDK also adds the `ClientFacingLabAccount` model for lab account management functionality.
* New `additionalProperty()` and `additionalProperties()` methods are now available on data model builders, providing better support for handling additional fields during serialization. The `ClientFacingMenstrualCycleSourceType` enum also includes a new `INSULIN_PUMP` value.
* The SDK now supports recovery readiness score data in sleep objects, providing a 0-100 value representing the provider's recovery/readiness proxy from Oura, Whoop, and Ultrahuman devices. Builder patterns across multiple types now support additional properties for enhanced extensibility.
* New builder methods available for setting additional properties on client-facing types. Use `additionalProperty(key, value)` to set individual custom fields or `additionalProperties(map)` to set multiple properties at once when constructing API objects.
* New `additionalProperty()` and `additionalProperties()` methods are now available on all builder classes. These methods allow setting additional properties dynamically when constructing objects, providing better support for extensible API responses.
* The SDK now supports compendium lab test conversion with the new `ConvertCompendiumResponse` type and `CompendiumSearchLabs` enum. Additional properties support has been standardized across all model builders with `additionalProperty()` and `additionalProperties()` methods.
* New builder methods available for setting additional properties on response objects. Use `additionalProperty(key, value)` or `additionalProperties(map)` on builders to add custom key-value pairs for extensibility. Also added new `GetTeamLabAccountsResponse` type.
* The SDK now supports adding additional properties to all Grouped* types through new builder methods. Use `additionalProperty(key, value)` to add individual properties or `additionalProperties(map)` to add multiple properties at once.
* New builder methods for adding custom properties to grouped data types. The `additionalProperty()` and `additionalProperties()` methods are now available on all grouped type builders (GroupedIge, GroupedSteps, etc.) allowing dynamic addition of properties at build time.
* The SDK now provides enhanced builder pattern support with new `additionalProperty()` and `additionalProperties()` methods for handling dynamic fields. New support for insulin injection timeseries queries and lab account delegation flow states is also available.
* The SDK now provides builder methods for setting additional properties on all types. Use `additionalProperty(key, value)` or `additionalProperties(map)` on any builder to handle dynamic fields returned by the API that aren't explicitly defined in the type schema.
* New `PerLabCandidate` type available for representing lab test candidates with detailed marker information, LOINC codes, CPT codes, and confidence scoring. Enhanced builder pattern support added to multiple existing types.
