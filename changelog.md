## 2.0.0 - 2026-04-02
* Several breaking changes have been made in this release:
* The `parserCreateJob(InputStream, String, ...)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers should migrate to the `parserCreateJob(File, BodyCreateLabReportParserJob, ...)` overloads instead.
* The `INSULIN_INJECTION` enum value has been removed from `IntervalTimeseriesExprTimeseries`. Update any switch statements or references accordingly.
* New features: SDK request/response logging is now configurable via `ClientOptions.builder().logging(LogConfig)`. SSE streams now support event-level discriminated union parsing via `Stream.fromSseWithEventDiscrimination(...)`.
* Several breaking changes have been made to the lab tests and link APIs:
* The four `parserCreateJob(InputStream, ...)` overloads on `RawLabReportClient` have been removed. Callers must migrate to the `File`-based `parserCreateJob` overloads.
* `LabTestsGetPscAppointmentAvailabilityRequest.getLab()` now returns `AppointmentPscLabs` instead of `String`; builder callers must pass an `AppointmentPscLabs` enum value instead of a raw string.
* The `teamId` field (`LinkBulkExportRequestTeamId`) has been removed from `BulkExportConnectionsBody`; remove any calls to `.teamId(...)` on that builder.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* The `getTeamId()` method and corresponding `teamId()` builder steps have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers that set or read `teamId` on these types must remove those references.
* The `address` field in `UserInfoCreateRequest` has changed type from `Address` to `UserAddress`. Callers must update their code to pass a `UserAddress` instance instead of `Address`.
* New optional fields have also been added: `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions` on `ManualConnectionData`; and `accessNotes` on `Address`.
* The SDK now supports additional expression types in `AggregateExprArg`: `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`, each with corresponding `Visitor` methods. `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields with dedicated enum types (`ClientFacingInsulinInjectionSampleDeliveryMode`, `ClientFacingInsulinInjectionSampleDeliveryForm`, `ClientFacingInsulinInjectionSampleBolusPurpose`). `AppointmentRescheduleRequest` now accepts an optional `appointmentNotes` field, and a new `AppointmentPscLabs` enum is available.
* Several breaking changes have been made to order and lab result types:
* **`ClientFacingOrder`**: The `parentId` field and its getter (`getParentId()`) and builder methods have been removed. A new required `lastEvent` field of type `ClientFacingOrderEvent` has been added — existing builder call chains must now include a `.lastEvent(...)` step between `.updatedAt(...)` and `.hasAbn(...)`. An optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` (an enum) instead of `String`. Callers that stored or compared the result as a `String` must be updated.
* **`HistoricalPullCompleted`**: The `isFinal` field is no longer settable. The `IsFinalStage` builder step has been removed; `getIsFinal()` now always returns `true`. Builder call chains that included `.isFinal(...)` must be updated.
* **`LinkBulkExportRequestTeamId`** enum has been removed and replaced by the new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`).
* New additions include an optional `statusDetail` field on `ClientFacingOrderEvent` and an optional `orderTransaction` field on `LabResultsRaw`.
* Several breaking changes are included in this release:
* **`ParsingJob`**: The `getJobId()` method and `JobIdStage` builder interface have been removed. Update builder chains from `.id(...).jobId(...).status(...)` to `.id(...).status(...)`.
* **`Placeholder`**: The staged builder interfaces (`PlaceholderStage`, `_FinalStage`) and `placeholder(boolean)` builder method are removed; `builder()` now returns `Builder` directly. `getPlaceholder()` return type changed from primitive `boolean` to `Boolean`.
* **`QueryGroupByItem.Visitor`**: Three new required visitor methods have been added — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)` — which must be implemented by all existing `Visitor` implementations.
* New optional `accessNotes` field has been added to `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* Several breaking changes are included in this release:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update references and builder calls from `Address` to `UserAddress`.
* **`UserRefreshSuccessResponse` and `UserRefreshErrorResponse`**: the `SuccessStage` builder interface and `success(boolean)` builder method have been removed. `builder()` now returns `UserIdStage` directly. The `getSuccess()` method now returns a hardcoded `Boolean` constant (`true`/`false`) rather than a stored field.
* **`QuerySelectItem.Visitor`** has three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. All existing `Visitor` implementations must add these methods to avoid compile errors.
* **`LinkBulkImportRequestTeamId`** (package `com.vital.api.resources.link.types`) has been replaced by `ResultMetadataGender` (package `com.vital.api.types`) with different enum values. **`LinkListBulkOpsRequestTeamId`** has been replaced by `SearchMode`.
* New additions include the `accessNotes` optional field on `UsAddress` and the `ResultMetadataGender` and `SearchMode` enums.
* The `getMarkers` methods now accept an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing callers to scope marker results to a specific lab by slug. Javadoc for `getMarkers` and `getMarkersForLabTest` overloads has been updated to accurately describe their behavior, including the note that reflex markers are excluded from `getMarkers` results.
* The `getMarkers` method on `LabTestsClient` now accepts an optional `lab_slug` filter parameter, allowing callers to narrow marker results to a specific lab by its slug. Javadoc has also been added to the `getMarkersForLabTest` methods to clarify that they return all markers for a lab test including reflex markers.
* The link provider manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, enabling mobile SDK version tracking. Additionally, the `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`); consumers relying on this parameter for those endpoints should remove it from their requests.
* The SDK now supports configurable HTTP-level logging via a new `LogConfig` class, which can be passed to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()`. Two new resource clients are available — `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`) — along with a `SseEvent<T>` type for Server-Sent Event handling. Several enums have been expanded with new values including `SONORA_QUEST` in `Labs`, `AppointmentProvider`, and `ClientFacingLabs`; `SAMSUNG_HEALTH` in `ManualProviders`; `TANDEM_SOURCE` in `PasswordProviders`; `APERO` and `PVERIFY` in `PayorCodeExternalProvider`; `CRL` in `Labs`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; and several new corrected/blocked status values in `OrderStatus`.
* The SDK now includes a new `CompendiumClient` (and async counterpart) with `search()` and `convert()` methods for the Compendium API. A new `SseEventParser` utility class has been added to support parsing Server-Sent Events with discriminated unions. Request builder classes such as `ActivityGetRequest`, `BodyGetRequest`, and `QueryBatch` now expose `additionalProperty()` and `additionalProperties()` builder methods for passing arbitrary extra fields.
* The SDK now includes a new `LabAccountClient` (and async variant) exposing a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request types, `ConvertCompendiumBody` and `SearchCompendiumBody`, are also available for compendium search and conversion operations. Several existing request builders have been extended with `additionalProperty` and `additionalProperties` methods for passing custom fields.
* The SDK now includes a `RawLabAccountClient` for direct HTTP access to lab account endpoints, supporting `getTeamLabAccounts()` with optional filtering by `labAccountId` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builders now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value data.
* All request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These optional builder methods allow SDK consumers to attach arbitrary extra fields to request payloads, providing forward compatibility with new API fields not yet modeled in the SDK.
* All request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary key-value pairs to requests. These methods are available on both the concrete `Builder` class and the `_FinalStage` interface for staged builders.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. This allows consumers to attach arbitrary extra properties to any Vitals request without requiring a new SDK release.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementation. This allows SDK consumers to attach arbitrary extra properties to vitals requests across all supported vitals endpoints.
* New `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods are now available on all request and type builder classes, allowing consumers to attach arbitrary extra fields to any constructed object. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for lab test canonical matching, exposing fields such as `loincSetHash`, `displayName`, `popularityScore`, and `confidence`. `ClientFacingAppointment` gains a new optional `appointmentNotes` field. All model builders now support `additionalProperty` and `additionalProperties` methods for setting arbitrary extra properties.
* Builder classes for `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface and the concrete `Builder` class. These new methods allow SDK consumers to attach arbitrary extra properties when constructing model objects using the builder pattern.
* Builder objects for all `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods via the `_FinalStage` interface, enabling consumers to attach arbitrary metadata during object construction. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* Builder classes for numerous `ClientFacing*` types (including Electrocardiogram, Fall, Fat, FloorsClimbed, Food, ForcedExpiratoryVolume1, ForcedVitalCapacity, Glucose, Handwashing, and HeartRate) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, making it easier to attach custom metadata when constructing objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* Builder instances for `ClientFacing*` types (including `ClientFacingHeartRateAlertChanged`, `ClientFacingHrvTimeseries`, `ClientFacingInsulinInjectionChanged`, `ClientFacingInsurance`, `ClientFacingLab`, and many more) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields during object construction. Two new enum types are also available: `ClientFacingInsulinInjectionSampleBolusPurpose` (MEAL, CORRECTION, MIXED, UNKNOWN) and `ClientFacingInsulinInjectionSampleDeliveryMode` (BASAL, BOLUS).
* The SDK now includes three new types: `ClientFacingLabAccount` (representing a lab account with billing and team allowlist details), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (webhook event types for lab report parsing jobs). Several existing builder types also gain `additionalProperty` and `additionalProperties` methods for attaching custom metadata.

The `ClientFacingSleepCycleSourceType` enum gains a new `INSULIN_PUMP` value, and builder types across many `ClientFacing*` classes now expose `additionalProperty` and `additionalProperties` methods for attaching arbitrary extra data during object construction.
