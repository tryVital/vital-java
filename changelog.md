## 2.0.0 - 2026-04-01
* **Breaking changes:**
* The `INSULIN_INJECTION` constant has been removed from the `IntervalTimeseriesExprTimeseries` enum. Any code referencing this value must be updated.
* The `parserCreateJob(InputStream, String)`, `parserCreateJob(InputStream, String, MediaType)`, `parserCreateJob(InputStream, String, RequestOptions)`, and `parserCreateJob(InputStream, String, MediaType, RequestOptions)` overloads have been removed from both `LabReportClient` and `AsyncLabReportClient`. Migrate to the equivalent `File`-based overloads.
**New features:**
* The SDK now supports configurable HTTP request/response logging. Pass a `LogConfig` instance to `ClientOptions.Builder.logging(...)` to enable log output; the SDK is silent by default.
* SSE streams with event-level discriminated unions are now supported via new `Stream.fromSseWithEventDiscrimination(...)` factory methods.
* Several breaking changes have been made to request model types:
* The four `parserCreateJob(InputStream, ...)` overloads on `RawLabReportClient` (and the raw variant) have been removed; callers must migrate to the `File`-based overloads instead.
* The `lab` parameter in `LabTestsGetPscAppointmentAvailabilityRequest` has changed from `String` to `AppointmentPscLabs`; callers must pass an `AppointmentPscLabs` enum value.
* The `teamId` field (`LinkBulkExportRequestTeamId`) has been removed from `BulkExportConnectionsBody`; callers relying on `getTeamId()` or the `teamId(...)` builder method must remove those usages.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes:
* The `getTeamId()` method and `teamId()` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers that set or read `teamId` on these objects must remove those calls.
* `UserInfoCreateRequest` now uses `UserAddress` instead of `Address` for its `address` field. The `AddressStage.address()` builder method now requires a `UserAddress` argument; callers must replace `Address` instances with `UserAddress`.
* Additionally, `ManualConnectionData` gains optional `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions` fields, and `Address` gains an optional `accessNotes` field. All modified builders now expose `additionalProperty` and `additionalProperties` helper methods.
* The `AggregateExprArg` union now supports three additional expression types: `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, and `InsulinInjectionTimeseriesExpr`. Existing implementations of the `AggregateExprArg.Visitor` interface must be updated to add handler methods for these new variants. `ClientFacingInsulinInjectionSample` gains optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields, and `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field.
* Several breaking changes have been made to order-related types in this release:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)`, `parentId(Optional<String>)`, `getParentId()`) have been removed. A new required builder stage `LastEventStage` has been inserted between `updatedAt(...)` and `hasAbn(...)`, meaning existing builder call chains must be updated to include a `lastEvent(ClientFacingOrderEvent)` call. New optional fields `clinicalNotes` and `lastEvent` have been added.
* **`GetOrderTransactionResponse`**: The `status` field type has changed from `String` to `OrderTransactionStatus`. Callers of `getStatus()` or the `status(String)` builder method must update to use `OrderTransactionStatus`.
* **`HistoricalPullCompleted`**: The `isFinal` constructor and builder parameter has been removed along with the `IsFinalStage` interface. The `getIsFinal()` method now always returns `true`. Any builder chains calling `.isFinal(...)` must be updated.
* **`ClientFacingOrderEvent`**: New optional `statusDetail` field added (backward-compatible addition).
* **`LabResultsRaw`**: New optional `orderTransaction` field added (backward-compatible addition).
* New `LabAccountStatus` enum added with values `ACTIVE`, `PENDING`, and `SUSPENDED`.
* The `ParsingJob` type no longer exposes a `jobId` field — the `getJobId()` method and the `JobIdStage` builder stage have been removed. Callers that reference `getJobId()` or chain `.jobId(…)` in the builder must be updated to remove those calls.
* The `QueryGroupByItem.Visitor` interface now declares three new `visit()` methods (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`); any exhaustive `Visitor` implementation must add these methods to compile.
* The `Placeholder` builder no longer exposes `PlaceholderStage` or `_FinalStage` interfaces, and the required `.placeholder(boolean)` builder step has been removed; callers should update to call `Placeholder.builder().build()` directly.
* Several breaking changes have been made to model types in this release.
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Any code calling `userInfo.getAddress()` or using the `AddressStage` builder step must be updated to use `UserAddress`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** builders no longer have a `success(boolean)` stage. The `SuccessStage` interface has been removed; builder construction now starts directly from `UserIdStage`. Code calling `.builder().success(...)` must be updated.
* **`QuerySelectItem.Visitor`** has two new required methods (`visit(DerivedReadinessColumnExpr)` and `visit(AwakeningsValueMacroExpr)` and `visit(InsulinInjectionTimeseriesExpr)`). Any exhaustive `Visitor` implementation must add these three new `visit` overloads.
* New additions include the `ResultMetadataGender` and `SearchMode` enums, the `UsAddress.accessNotes` optional field, and three new `QuerySelectItem` expression types.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing callers to scope marker results to a specific lab by its slug. Javadoc has also been added to all `getMarkersForLabTest` overloads describing their behavior with reflex markers.
* The `getMarkers` endpoint on `LabTestsClient` now supports an optional `lab_slug` query parameter, allowing callers to filter markers by a specific lab's slug. The Javadoc for `getMarkers` and `getMarkersForLabTest` has also been updated to clarify which markers are included or excluded (e.g., reflex markers).
* The SDK now supports sending optional mobile SDK version headers (`x-vital-ios-sdk-version` and `x-vital-android-sdk-version`) when calling the manual provider link endpoint. Populate the `vitalIosSdkVersion` and `vitalAndroidSdkVersion` fields on the request object to include these headers.
* The SDK now supports configurable request/response logging via a new `LogConfig` API. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable log output at your chosen level; logging remains silent by default.
* Two new resource clients are now available: `vital.compendium()` (`CompendiumClient`) and `vital.labAccount()` (`LabAccountClient`), accessible from both the sync `Vital` and async `AsyncVital` top-level clients.
* Several enums have been extended with new values: `Labs` gains `SONORA_QUEST` and `CRL`; `AppointmentProvider` and `ClientFacingLabs` gain `SONORA_QUEST`; `ManualProviders` gains `SAMSUNG_HEALTH`; `PasswordProviders` gains `TANDEM_SOURCE`; `PayorCodeExternalProvider` gains `APERO` and `PVERIFY`; `OrderLowLevelStatus` gains `CORRECTED` and `LAB_PROCESSING_BLOCKED`; and `OrderStatus` gains `CORRECTED` statuses for walk-in, at-home phlebotomy, on-site collection, and testkit, plus `SAMPLE_WITH_LAB_TESTKIT_LAB_PROCESSING_BLOCKED`.
* The SDK now includes a `CompendiumClient` (and async variant) with `search()` and `convert()` methods for interacting with the `v3/compendium` endpoints. Several request builder classes (`ActivityGetRequest`, `ActivityGetRawRequest`, `BodyGetRequest`, `BodyGetRawRequest`, `QueryBatch`, `AggregateGetTaskHistoryForContinuousQueryRequest`) now expose `additionalProperty()` and `additionalProperties()` builder methods for passing arbitrary extra fields. A new `SseEventParser` utility has also been added to support Server-Sent Events with discriminated union deserialization.
* The SDK now includes a new `LabAccountClient` (and async equivalent) with a `getTeamLabAccounts()` method for retrieving team lab account information. Two new compendium request types are also available: `SearchCompendiumBody` for searching the lab compendium and `ConvertCompendiumBody` for converting between labs. Several existing request builders now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* The SDK now includes a new `RawLabAccountClient` for retrieving team lab accounts with optional filtering by `lab_account_id` and `status`. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test and lab report request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing extra parameters.
* Request builder classes across all resource packages now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new builder methods allow SDK consumers to pass arbitrary extra fields when constructing request objects, improving forward compatibility with evolving API payloads.
* All user and vitals request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary properties to requests without requiring SDK updates.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementation. These new methods allow consumers to attach arbitrary key-value metadata to request objects, improving extensibility across all Vitals request types.
* Vitals request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on the `_FinalStage` interface. This allows consumers to attach arbitrary key-value pairs to requests for forward-compatibility with API parameters not yet modeled in the SDK.
* The SDK now supports setting arbitrary additional properties on request and type builders via new `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods across all affected classes. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now includes an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type to support lab test canonicalization, with fields for LOINC set hash, display name, aliases, LOINC/CPT codes, provider IDs, and scoring. An optional `appointmentNotes` field has been added to `ClientFacingAppointment`. Additionally, builder interfaces across many response and event types now expose `additionalProperty` and `additionalProperties` methods, allowing consumers to attach arbitrary metadata when constructing objects programmatically.
* The builder for every `ClientFacing*` type now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary metadata to these objects at build time.
* Builder objects for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface, allowing consumers to attach arbitrary extra properties when constructing SDK model objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* Builder instances for many `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, making it easier to attach custom metadata when constructing objects. The `ClientFacingElectrocardiogramSourceType` enum also gains a new `INSULIN_PUMP` value to represent insulin pump data sources.
* The SDK now exposes `additionalProperty` and `additionalProperties` builder methods on `ClientFacing*` type builders, allowing consumers to attach arbitrary extra fields when constructing these objects. Two new enums have also been added: `ClientFacingInsulinInjectionSampleBolusPurpose` (with values `MEAL`, `CORRECTION`, `MIXED`, `UNKNOWN`) and `ClientFacingInsulinInjectionSampleDeliveryMode` (with values `BASAL`, `BOLUS`) to support insulin injection sample metadata.
* The SDK now includes three new types: `ClientFacingLabAccount` (for lab account management), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (for lab report parsing job webhook events). Several existing builder types also now expose `additionalProperty` and `additionalProperties` methods on their `_FinalStage` interfaces, enabling consumers to attach arbitrary extra properties when constructing those objects.
* Builder instances for many `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easy to attach arbitrary extra fields when constructing these objects. The `ClientFacingMenstrualCycleSourceType` enum also gains a new `INSULIN_PUMP` value.
* The `ClientFacingSleep` type now includes a new optional `recoveryReadinessScore` field (a 0–100 scalar sourced from Oura readiness score, Whoop recovery score, and Ultrahuman recovery). A new `INSULIN_PUMP` value has been added to the `ClientFacingSleepCycleSourceType` enum. Builder classes across many `ClientFacing*` types now expose `additionalProperty` and `additionalProperties` methods for attaching arbitrary metadata.

This covers a wide range of types including event, timeseries, sample, team, user, and order detail models.
