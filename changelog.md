## 2.0.0 - 2026-04-09
* The `INSULIN_INJECTION` enum value has been removed from `IntervalTimeseriesExprTimeseries`. Update any code referencing this value or switch statements that include it.
* The `parserCreateJob` overloads accepting `InputStream` have been removed from `LabReportClient` and `AsyncLabReportClient`. Migrate to the equivalent `File`-based overloads (`parserCreateJob(File, BodyCreateLabReportParserJob)`).
* The SDK now supports configurable logging via a new `logging(LogConfig)` builder option on `ClientOptions`, and `Stream` gains SSE event-level discrimination support through new `fromSseWithEventDiscrimination` factory methods.
* Several breaking changes have been made to lab-related and link request classes:
* **`LabTestsGetPscAppointmentAvailabilityRequest.lab`** now requires `AppointmentPscLabs` (a typed enum) instead of a plain `String`. Update all builder calls from `.lab("quest")` to `.lab(AppointmentPscLabs.QUEST)` (or the appropriate enum value).
* **`RawLabReportClient.parserCreateJob(InputStream, ...)`** overloads have been removed. Replace them with the equivalent `File`-based `parserCreateJob(File, ...)` overloads.
* **`BulkExportConnectionsBody.teamId` / `getTeamId()`** has been removed. Remove any calls to `teamId(...)` on the builder and `getTeamId()` on the built object.
* New optional fields have also been added: `labAccountId` and `labSlug` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request and type classes in this release:
* The `getTeamId()` method and `teamId()` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. Callers that set or read `teamId` on these objects must remove those usages.
* The `address` field in `UserInfoCreateRequest` has changed its type from `Address` to `UserAddress`. Callers must update their code to pass a `UserAddress` instance instead of an `Address` instance when building `UserInfoCreateRequest`.
* New optional fields have also been added: `ManualConnectionData` gains `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` gains `accessNotes`.
* The `AggregateExprArg.Visitor<T>` interface now requires three new method implementations: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing this interface must add these methods or it will fail to compile. Additionally, `LinkBulkPauseRequestTeamId` and `LinkBulkTriggerHistoricalPullRequestTeamId` have been removed and replaced by `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm` respectively.
* New optional fields have been added to `ClientFacingInsulinInjectionSample` (`deliveryMode`, `deliveryForm`, `bolusPurpose`) and `AppointmentRescheduleRequest` (`appointmentNotes`), along with new supporting enum types.
* Several breaking changes have been made to order-related types in this release:
* **`ClientFacingOrder`**: The `parentId` field and its getter (`getParentId()`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) and optional `clinicalNotes` field have been added. The builder stage chain has changed — `UpdatedAtStage` now returns `LastEventStage` instead of `HasAbnStage`, so existing builder chains must call `.lastEvent(...)` before `.hasAbn(...)`.
* **`GetOrderTransactionResponse`**: The `getStatus()` method now returns `OrderTransactionStatus` (an enum) instead of `String`. Update any code that assigns or compares the result of `getStatus()` to a plain `String`.
* **`HistoricalPullCompleted`**: The `isFinal` builder parameter has been removed (`IsFinalStage` interface no longer exists). `getIsFinal()` now always returns `true`. Remove any calls to `.isFinal(...)` in builder chains.
* New additions include an optional `statusDetail` field on `ClientFacingOrderEvent`, an optional `orderTransaction` field on `LabResultsRaw`, and a new `LabAccountStatus` enum.
* Several breaking changes are included in this release:
* **`ParsingJob`**: The `getJobId()` method and its associated `JobIdStage` builder interface have been removed. Update any code calling `parsingJob.getJobId()` or using `.jobId(...)` in the builder chain.
* **`Placeholder`**: The required `placeholder` field has been removed from the builder. `Placeholder.builder()` now returns a `Builder` directly (the `PlaceholderStage` interface is gone). `getPlaceholder()` now always returns `true`.
* **`QueryGroupByItem.Visitor`**: Three new `visit` methods are now required — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. All exhaustive `Visitor` implementations must be updated.
* New additions (backward-compatible): `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now expose an optional `accessNotes` field via `getAccessNotes()` and a corresponding builder method.
* Several breaking changes have been introduced in this release:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update their type references accordingly.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse`** builders no longer require a `success(boolean)` step — the `SuccessStage` interface has been removed. Update builder chains by removing the `.success(...)` call. `getSuccess()` now returns a boxed `Boolean` with a hardcoded constant value (`false` / `true`) instead of reading a stored field.
* **`QuerySelectItem.Visitor`** gains three new required methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any class implementing `Visitor` must add these methods.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** have been removed. New enums `ResultMetadataGender` and `SearchMode` are available in `com.vital.api.types`.
* New additions: `UsAddress` now supports an optional `accessNotes` field, and new types `DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`, `ResultMetadataGender`, and `SearchMode` have been added.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`. Javadoc has been added to all `getMarkersForLabTest` overloads clarifying that reflex markers are included, and the `getMarkers` documentation now notes that reflex markers are not included.
* The `getMarkers` method on `LabTestsClient` now accepts a `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing consumers to filter markers by lab slug. Javadoc has also been added to all `getMarkersForLabTest` overloads to clarify that reflex markers are included in those results.
* The SDK now sends optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` HTTP headers when calling the link provider manual endpoint, enabling mobile SDK version tracking. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`) — callers relying on this parameter to filter by team should be aware it will no longer be sent.
* The SDK now supports configurable logging via a new `LogConfig` class. Pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable request/response logging with customizable log levels and logger implementations (silent by default).
* Two new resource clients are available: `CompendiumClient` (accessible via `vital.compendium()`) and `LabAccountClient` (accessible via `vital.labAccount()`), along with their async counterparts.
* Several enums have been expanded with new values: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; multiple `CORRECTED` and `LAB_PROCESSING_BLOCKED` statuses in `OrderStatus`; `TANDEM_SOURCE` in `PasswordProviders`; and `APERO` and `PVERIFY` in `PayorCodeExternalProvider`.
* The SDK now includes a new `CompendiumClient` (and async variant `AsyncCompendiumClient`) with `search()` and `convert()` methods for the Vital Compendium API. Request builder classes across multiple resources now expose `additionalProperty()` and `additionalProperties()` methods, allowing consumers to attach arbitrary extra fields to requests. A new internal `SseEventParser` utility also enables structured handling of Server-Sent Events with discriminated union support.
* The SDK now includes a new `LabAccountClient` (and async counterpart) exposing a `getTeamLabAccounts()` method for retrieving team lab accounts. Two new compendium request types, `SearchCompendiumBody` and `ConvertCompendiumBody`, are also available for building compendium search and conversion requests. Additionally, several existing request builders now expose `additionalProperty` and `additionalProperties` helper methods for passing through custom request metadata.
* The SDK now includes a new `RawLabAccountClient` that provides raw HTTP access to the lab account endpoint (`getTeamLabAccounts`), allowing filtering by `lab_account_id` and `status`. An optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible` for passing clinical notes when creating orders. Additionally, all lab test and lab report request builders now expose `additionalProperty` and `additionalProperties` methods for setting arbitrary extra properties.
* Builder classes for all request and payload types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra fields to any request object, providing forward-compatibility with API fields not yet modeled in the SDK.
* All request builder classes now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra properties to requests for forward-compatibility. These methods are available on both the `Builder` and `_FinalStage` interfaces across all user and vitals request types.
* The vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. These new methods allow SDK consumers to attach arbitrary key-value pairs to vitals requests, enabling forward-compatible extension without requiring an SDK update for each new field.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface, allowing consumers to attach arbitrary key-value metadata to requests. This is available across all affected `Vitals*Request` and `Vitals*GroupedRequest` classes.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on all request and model builder classes, making it easy to include extra fields in serialized payloads. A new `AwakeningsValueMacroExpr` type has been added, and `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type representing a lab test canonical candidate with LOINC, CPT, and provider metadata. All response object builders now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on their `_FinalStage` interfaces, enabling programmatic injection of extra fields. `ClientFacingAppointment` has a new optional `appointmentNotes` field accessible via `getAppointmentNotes()`.
* Builder instances for all `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields when constructing these objects. These methods are also declared on each type's `_FinalStage` builder interface.
* The SDK now supports setting arbitrary additional properties on `ClientFacing*` model builders via new `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* Builder objects for numerous `ClientFacing*` types (electrocardiogram, fall, fat, floors climbed, food, forced expiratory volume, forced vital capacity, glucose, handwashing, heart rate, and more) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary metadata when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* The SDK now supports setting arbitrary additional properties on `ClientFacing*` builder types via new `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on `_FinalStage`. Two new enums, `ClientFacingInsulinInjectionSampleBolusPurpose` and `ClientFacingInsulinInjectionSampleDeliveryMode`, have been added to support insulin injection sample data.
* The SDK now includes three new types: `ClientFacingLabAccount` (representing a lab account with billing and team allowlist details), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (webhook event types for lab report parsing job lifecycle). Several existing builder interfaces also gain new `additionalProperty` and `additionalProperties` methods for greater extensibility.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` builder methods on over 25 `ClientFacing*` types, allowing consumers to attach arbitrary extra fields through the fluent staged-builder API. A new `INSULIN_PUMP` value has also been added to the `ClientFacingMenstrualCycleSourceType` enum.
* The `ClientFacingSleep` type now exposes an optional `recoveryReadinessScore` field (0–100), aggregating recovery/readiness scores from Oura, Whoop, and Ultrahuman. The `ClientFacingSleepCycleSourceType` enum gains a new `INSULIN_PUMP` value. Additionally, all affected builder types now support `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods for attaching extra metadata when constructing request or response objects.
* Builder classes for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary custom properties when constructing objects via the builder pattern.
* Builder instances for `ClientFacing*` types (including water, weight, wheelchair push, workout variants, walk-in test order, and others) as well as `ClientSleepCycleResponse`, `ClientSleepResponse`, `ClientWorkoutResponse`, `ClinicalInformation`, `ClientUserIdConflict`, `ClientUserIdConflictResponse`, and `ClientFacingWorkoutMap` now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary extra fields when constructing objects via the builder pattern.
* The SDK now includes three new types: `CompendiumSearchLabs` (an enum of supported lab identifiers), `ConvertCompendiumResponse` (representing compendium conversion results with provider ID mappings), and `DerivedReadinessColumnExpr` / `DerivedReadinessColumnExprDerivedReadiness` (for readiness-derived column expressions). Additionally, numerous existing builder types now expose `additionalProperty` and `additionalProperties` methods on their `_FinalStage` interfaces, making it easier to attach arbitrary metadata when constructing objects.
