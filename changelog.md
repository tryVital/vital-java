## 2.0.0 - 2026-04-03
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream stream, String filename, ...)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Callers must migrate to the `File`-based overloads: `parserCreateJob(File file, BodyCreateLabReportParserJob request)`.
* The `INSULIN_INJECTION` enum value has been removed from `IntervalTimeseriesExprTimeseries`; any exhaustive switch statements referencing this value must be updated.
* New capabilities also ship in this release: the SDK now supports configurable logging via the `ClientOptions.Builder.logging(LogConfig)` method (silent by default), and SSE streams gain event-level discrimination support through new `Stream.fromSseWithEventDiscrimination(...)` factory methods.
* Several lab test request types have been updated with new optional fields and breaking API changes:
* `LabTestsGetPscAppointmentAvailabilityRequest.lab` now requires an `AppointmentPscLabs` enum value instead of a plain `String`; callers must update to pass the appropriate `AppointmentPscLabs` constant.
* `BulkExportConnectionsBody` no longer exposes the `teamId` field or its builder methods (`teamId(...)`) — callers referencing `getTeamId()` or `.teamId(...)` must remove those calls.
* `RawLabReportClient` no longer provides `parserCreateJob(InputStream, ...)` overloads; callers should switch to the `File`-based overloads.
* New optional fields `labAccountId` and `labSlug` are available on `CreateLabTestRequest`, `labSlug` is available on `LabTestsGetMarkersRequest`, and `appointmentNotes` is available on `RequestAppointmentRequest`.
* The `teamId` field and its associated getter/builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest`. The `address` field type in `UserInfoCreateRequest` has changed from `Address` to `UserAddress` — callers must update their imports and usages accordingly. `ManualConnectionData` gains optional `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions` fields; `Address` gains an optional `accessNotes` field.
* The `AggregateExprArg.Visitor` interface now requires three additional methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this visitor interface will fail to compile until these methods are added.
* Additionally, `ClientFacingInsulinInjectionSample` gains optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields, `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field, and new enums `AppointmentPscLabs` and `ClientFacingInsulinInjectionSampleDeliveryForm` are available.
* Several breaking changes have been made to order-related types in this release:
* **`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)`, `parentId(Optional<String>)`) have been removed. Additionally, a new required `lastEvent` field (`ClientFacingOrderEvent`) has been inserted into the builder chain — existing builder call chains will fail to compile without adding `.lastEvent(...)` between `.updatedAt(...)` and subsequent steps. A new optional `clinicalNotes` field has also been added.
* **`GetOrderTransactionResponse`**: The return type of `getStatus()` has changed from `String` to `OrderTransactionStatus`. Callers that assign the result to a `String` variable must be updated.
* **`HistoricalPullCompleted`**: The `isFinal` builder parameter has been removed. The `IsFinalStage` interface no longer exists; existing builder chains calling `.isFinal(...)` will not compile. `getIsFinal()` now always returns `true`.
* **`LinkBulkExportRequestTeamId`** enum has been removed and replaced by the new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`).
* **`ClientFacingOrderEvent`** and **`LabResultsRaw`** gain new optional fields (`statusDetail` and `orderTransaction`, respectively) that are backward compatible.
* Several breaking changes have been made to SDK types in this release:
* **`ParsingJob`**: The `getJobId()` method and `JobIdStage` builder interface have been removed. Remove any `.jobId(...)` calls from builder chains — `.id(...)` now advances directly to `StatusStage`.
* **`QueryGroupByItem.Visitor`**: Three new `visit` methods are required — `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Existing `Visitor` implementations must be updated to implement these methods.
* **`Placeholder`**: The `PlaceholderStage` builder interface and required `placeholder(boolean)` builder step have been removed; `builder()` now returns `Builder` directly and `getPlaceholder()` always returns `true`.
* Additionally, `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation` now support an optional `accessNotes` field accessible via `getAccessNotes()` and settable via the builder.
* Several breaking changes have been made to the SDK's public API surface:
* **`UserInfo.getAddress()`** now returns `UserAddress` instead of `Address`. Callers must update their type references and any code that passes an `Address` to `UserInfo.Builder.address()`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse` builders**: the required `SuccessStage` has been removed from the staged builder chain. Existing code that calls `.success(true/false)` on these builders will fail to compile — remove that call from the chain (the value is now hardcoded).
* **`QuerySelectItem.Visitor`**: three new required visit methods (`visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, `visit(InsulinInjectionTimeseriesExpr)`) have been added to the `Visitor` interface. All existing `Visitor` implementations must implement these new methods.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** have been removed and replaced by `ResultMetadataGender` and `SearchMode` respectively.
* New additions include the `accessNotes` optional field on `UsAddress`, the `ResultMetadataGender` and `SearchMode` enums, and `additionalProperty`/`additionalProperties` builder helpers on several types.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter, allowing callers to narrow results to a specific lab by its slug. Javadoc has been added to all `getMarkersForLabTest` overloads clarifying that reflex markers are included, while `getMarkers` documentation now notes that reflex markers are excluded and only active, orderable markers are returned.
* The `getMarkers` method on the lab tests client now supports an optional `lab_slug` filter parameter, allowing callers to narrow marker results to a specific lab by slug. The `getMarkersForLabTest` methods now include Javadoc clarifying that reflex markers are included in results, while `getMarkers` Javadoc notes that reflex markers are excluded.
* The link provider manual connect endpoint now supports forwarding optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers when provided on the request object. Additionally, the `team_id` query parameter has been removed from several bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`).
* The SDK now supports configurable logging via a new `LogConfig` API — pass a `LogConfig` instance to `VitalBuilder.logging()` or `AsyncVitalBuilder.logging()` to enable HTTP request/response logging with customizable log levels and logger implementations (silent by default).
* Two new resource clients are available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async counterparts.
* New enum values have been added: `SONORA_QUEST` in `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` in `Labs`; `SAMSUNG_HEALTH` in `ManualProviders`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` in `OrderLowLevelStatus`; five new corrected/blocked statuses in `OrderStatus`; `TANDEM_SOURCE` in `PasswordProviders`; and `APERO` and `PVERIFY` in `PayorCodeExternalProvider`. Exhaustive `switch` statements over these enums will require updates to handle the new cases.
* The SDK now includes a new `CompendiumClient` (and `AsyncCompendiumClient`) with `search()` and `convert()` methods for interacting with the v3 Compendium API. A new `SseEventParser` utility class has also been added to support parsing Server-Sent Events with discriminated unions. Request builder classes across multiple resources now support `additionalProperty` and `additionalProperties` methods for passing arbitrary extra fields.
* The SDK now includes a new `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab account information. Two new compendium request types, `ConvertCompendiumBody` and `SearchCompendiumBody`, are also available for compendium conversion and search operations. Several existing request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing custom key-value pairs.
* The SDK now includes a new `RawLabAccountClient` that exposes a `getTeamLabAccounts()` method for retrieving lab accounts with optional `lab_account_id` and `status` filters. A new optional `clinicalNotes` field has been added to `CreateOrderRequestCompatible`. All lab test request builder classes now expose `additionalProperty` and `additionalProperties` methods for passing through custom properties.
* Request builder classes across the SDK now support `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to API request objects. These methods are available on the builder's `_FinalStage` interface where staged builders are used.
* All request builder classes across the `user` and `vitals` resource packages now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These new methods allow SDK consumers to attach arbitrary extra properties to any request object using the fluent builder pattern.
* Vitals request builders now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. These new methods are available across all vitals request types (cholesterol, distance, glucose, heart rate, HRV, hypnogram, IGE, and more), allowing consumers to attach arbitrary extra key-value pairs to any request object.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface, enabling consumers to attach arbitrary extra parameters to vitals requests. This addition is backward-compatible and affects all Vitals request and grouped request types.
* The SDK now exposes `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` builder methods on request and type classes, allowing consumers to pass arbitrary extra properties. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type representing LOINC/CPT candidate data with fields such as `loincSetHash`, `displayName`, `popularityScore`, and `confidence`. `ClientFacingAppointment` gains an optional `appointmentNotes` field. All affected builder types now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods on both the `_FinalStage` interface and the concrete `Builder`, making it easier to attach arbitrary extra properties to request and response objects.
* Builder classes for all `ClientFacing*` model types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow SDK consumers to attach arbitrary extra fields to any model object at build time, providing a forward-compatible way to pass properties not yet formally modelled in the SDK.
* Builder objects for `ClientFacing*` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary metadata without requiring SDK updates. A new `INSULIN_PUMP` value has also been added to the `ClientFacingDeviceSourceType` enum.
* The SDK now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object>)` builder methods on more than 30 `ClientFacing*` types, allowing consumers to attach arbitrary properties when constructing these objects. A new `INSULIN_PUMP` value has also been added to the `ClientFacingElectrocardiogramSourceType` enum.
* The builder interfaces for numerous `ClientFacing*` types (including `ClientFacingHeartRateAlertChanged`, `ClientFacingHrvTimeseries`, `ClientFacingInsurance`, `ClientFacingLab`, and many more) now expose `additionalProperty(String, Object)` and `additionalProperties(Map<String, Object>)` methods, allowing consumers to attach arbitrary extra fields when constructing objects. Two new enums, `ClientFacingInsulinInjectionSampleBolusPurpose` and `ClientFacingInsulinInjectionSampleDeliveryMode`, have been added to support insulin injection sample metadata.
* The SDK now includes three new types: `ClientFacingLabAccount` (representing a lab account with billing, status, and team allowlist fields), `ClientFacingLabReportParsingJobCreatedEvent`, and `ClientFacingLabReportParsingJobUpdatedEvent` (for webhook events around lab report parsing jobs). Several existing builder interfaces also gain `additionalProperty` and `additionalProperties` methods for more flexible object construction.
* The builder for numerous `ClientFacing*` types (including `ClientFacingMealResponse`, `ClientFacingMenstrualCycle`, `ClientFacingProvider`, `ClientFacingPayor`, and 20+ others) now exposes `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easier to attach arbitrary metadata when constructing these objects. The `ClientFacingMenstrualCycleSourceType` enum also gains a new `INSULIN_PUMP` value.
* The `ClientFacingSleep` type now exposes a new optional `recoveryReadinessScore` field (0–100), representing the provider's recovery/readiness proxy sourced from Oura readiness score, Whoop recovery score, and Ultrahuman recovery. Builders across many response types (e.g. `ClientFacingSource`, `ClientFacingSleepCycle`, `ClientFacingShipment`, and others) now support `additionalProperty` and `additionalProperties` methods for attaching arbitrary metadata. A new `INSULIN_PUMP` value has also been added to the `ClientFacingSleepCycleSourceType` enum.
* Builder classes for all `ClientFacing*` types now expose two new fluent methods — `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` — that allow consumers to attach arbitrary extra fields when constructing objects. Existing code is unaffected; no changes are required.
* Builder classes for numerous SDK model types (including `ClientFacingWorkout`, `ClientFacingWaterTimeseries`, `ClientFacingWheelchairPushSample`, `ClientUserIdConflict`, `ClinicalInformation`, and many more) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow attaching arbitrary extra fields when constructing model objects without affecting existing builder usage.
* The SDK now includes three new types: `CompendiumSearchLabs` (an enum for supported lab values), `ConvertCompendiumResponse` (a response model for compendium conversions), and `DerivedReadinessColumnExpr` / `DerivedReadinessColumnExprDerivedReadiness` (for derived readiness column expressions). Additionally, builder classes across many existing types now expose `additionalProperty(String, Object)` and `additionalProperties(Map)` methods on their `_FinalStage` interfaces, allowing consumers to attach arbitrary extra JSON fields when constructing request and response objects.
* Builder objects across many SDK types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, making it easy to attach arbitrary extra fields when constructing request or response objects. A new `GetTeamLabAccountsResponse` type has also been added to support retrieving team lab account data.
* The builder classes for all `Grouped*` and `Grouped*Response` types (e.g. `GroupedHeartRate`, `GroupedGlucoseResponse`, etc.) now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. These allow consumers to attach arbitrary metadata when constructing these objects, improving forward compatibility with evolving API responses.
* The builders for `GroupedIge`, `GroupedIgg`, `GroupedInhalerUsage`, `GroupedInsulinInjection`, `GroupedLeanBodyMass`, `GroupedMindfulnessMinutes`, `GroupedNote`, `GroupedPeakExpiratoryFlowRate`, `GroupedRespiratoryRate`, `GroupedSleepApneaAlert`, `GroupedSleepBreathingDisturbance`, `GroupedStandDuration`, `GroupedStandHour`, `GroupedSteps`, `GroupedStressLevel`, `GroupedUvExposure`, `GroupedVo2Max`, and their corresponding `*Response` types now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods. New types have also been added: `InsulinInjectionTimeseriesExpr` and `InsulinInjectionTimeseriesExprField` for querying insulin injection data, and `LabAccountDelegatedFlow` enum for lab account delegation flows.
