## 2.0.0 - 2026-04-09
* Several breaking changes are included in this release:
* The `parserCreateJob(InputStream, ...)` overloads have been removed from `LabReportClient` and `AsyncLabReportClient`. Migrate to the `File`-based overloads: `parserCreateJob(File file, BodyCreateLabReportParserJob request)`.
* The `INSULIN_INJECTION` value has been removed from the `IntervalTimeseriesExprTimeseries` enum; any references to this constant must be removed.
* New in this release: the SDK now supports configurable request/response logging via `ClientOptions.builder().logging(LogConfig)`, and streaming now supports event-discriminated SSE payloads via `Stream.fromSseWithEventDiscrimination(...)`.
* Several breaking changes have been made to lab test and link request types:
* **`LabTestsGetPscAppointmentAvailabilityRequest`**: The `lab` field has changed type from `String` to `AppointmentPscLabs`. Update calls to `getLab()` and the builder's `lab(...)` method to use the new `AppointmentPscLabs` enum.
* **`BulkExportConnectionsBody`**: The `teamId` field and its builder methods (`teamId(...)`, `getTeamId()`) have been removed. Remove any references to these.
* **`RawLabReportClient`**: The four `parserCreateJob(InputStream, ...)` overloads have been removed. Use the existing `File`-based overloads instead.
* New optional fields have also been added: `labSlug` and `labAccountId` on `CreateLabTestRequest`, `labSlug` on `LabTestsGetMarkersRequest`, and `appointmentNotes` on `RequestAppointmentRequest`.
* Several breaking changes have been made to request types in this release.
* The `getTeamId()` method and `teamId(...)` builder methods have been removed from `BulkImportConnectionsBody`, `BulkPauseConnectionsBody`, `BulkTriggerHistoricalPullBody`, and `LinkListBulkOpsRequest` — callers using these methods must remove them. The `address` field in `UserInfoCreateRequest` has changed its type from `Address` to `UserAddress`; callers must update their code to pass a `UserAddress` instance instead.
* New optional fields have been added: `ManualConnectionData` now supports `vitalIosSdkVersion`, `vitalAndroidSdkVersion`, and `grantedPermissions`; `Address` now supports `accessNotes`.
* The `AggregateExprArg.Visitor` interface now requires three additional `visit()` methods: `visit(DerivedReadinessColumnExpr)`, `visit(AwakeningsValueMacroExpr)`, and `visit(InsulinInjectionTimeseriesExpr)`. Any existing implementation of this interface must be updated to implement these methods or a compile error will occur.
* Additionally, `ClientFacingInsulinInjectionSample` now exposes optional `deliveryMode`, `deliveryForm`, and `bolusPurpose` fields with new enums (`ClientFacingInsulinInjectionSampleDeliveryMode`, `ClientFacingInsulinInjectionSampleDeliveryForm`, `ClientFacingInsulinInjectionSampleBolusPurpose`). `AppointmentRescheduleRequest` gains an optional `appointmentNotes` field, and a new `AppointmentPscLabs` enum (`QUEST`, `SONORA_QUEST`) has been added.
* Several breaking changes have been made to order-related types in this release.
**`ClientFacingOrder`**: The `parentId` field and its builder methods (`parentId(String)`, `parentId(Optional<String>)`) have been removed. A new required `lastEvent` field (`ClientFacingOrderEvent`) has been added, inserting a new `LastEventStage` into the builder chain between `UpdatedAtStage` and `HasAbnStage` — existing builder call chains must be updated to call `.lastEvent(...)` after `.updatedAt(...)`. A new optional `clinicalNotes` field is also available.
**`GetOrderTransactionResponse`**: The `status` field and `StatusStage.status()` method parameter have changed from `String` to the new `OrderTransactionStatus` enum. Callers that pass a raw string or expect a `String` return value must migrate to the `OrderTransactionStatus` enum.
**`HistoricalPullCompleted`**: The `IsFinalStage` builder interface has been removed. Existing builder chains that call `.isFinal(boolean)` must remove that call; `getIsFinal()` now always returns `true` (as `Boolean`).
* A new `LabAccountStatus` enum (`ACTIVE`, `PENDING`, `SUSPENDED`) has been added. `ClientFacingOrderEvent` and `LabResultsRaw` have new optional fields (`statusDetail` and `orderTransaction` respectively).
* Several breaking changes have been made to model types in this release.
* **`ParsingJob`**: The `getJobId()` method and the `JobIdStage` builder interface have been removed. Code that calls `.jobId(...)` in the builder chain must be updated — the builder now proceeds directly from `.id(...)` to `.status(...)`.
* **`Placeholder`**: The `getPlaceholder()` method no longer returns a stored `boolean` value (it always returns `true`), and the staged `PlaceholderStage` / `_FinalStage` builder interfaces have been removed. The builder is now a plain `Builder` class without a required `placeholder(boolean)` step.
* **`QueryGroupByItem.Visitor`**: Three new `visit` methods have been added (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). Any exhaustive `Visitor` implementation must be updated to add these methods.
* New optional `accessNotes` field added to `PatientAddress`, `PatientAddressCompatible`, and `PatientAddressWithValidation`.
* Several breaking changes have been introduced in this release:
* **`QuerySelectItem.Visitor`** now requires three new `visit()` methods (`DerivedReadinessColumnExpr`, `AwakeningsValueMacroExpr`, `InsulinInjectionTimeseriesExpr`). Any class implementing `Visitor` must add these methods or it will fail to compile.
* **`UserInfo.address`** type has changed from `Address` to `UserAddress`. Callers that pass an `Address` to `.address(...)` or use `getAddress()` as an `Address` must update to `UserAddress`.
* **`UserRefreshErrorResponse` and `UserRefreshSuccessResponse` builders** no longer have a `SuccessStage`; the `success(boolean)` builder step has been removed and `getSuccess()` now returns a constant `Boolean`. Remove any `.success(...)` calls from builder chains.
* **`LinkBulkImportRequestTeamId`** and **`LinkListBulkOpsRequestTeamId`** enums have been removed and replaced by `ResultMetadataGender` and `SearchMode` respectively.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter via `LabTestsGetMarkersRequest`, allowing results to be scoped to a specific lab by slug. Javadoc has also been added to all `getMarkersForLabTest` overloads, clarifying that reflex markers are included in those results.
* The `getMarkers` method now supports an optional `lab_slug` filter parameter to narrow results to a specific lab. Javadoc on `getMarkers` has been updated to clarify that reflex markers are excluded, while `getMarkersForLabTest` now documents that reflex markers are included in its results. The SDK also fixes a potential issue where the `drawCompleted` PATCH request was sent without a required empty body.
* The Link provider manual endpoint now supports optional `x-vital-ios-sdk-version` and `x-vital-android-sdk-version` request headers, enabling mobile SDK consumers to pass their platform version when connecting a manual provider. The `team_id` query parameter has been removed from bulk link operations (`bulk_import`, `bulk_trigger_historical_pull`, `bulk_export`, `bulk_pause`) to align with the current API contract.
* The SDK now supports configurable HTTP request/response logging via a new `LogConfig` API — silent by default and opt-in via `VitalBuilder.logging(LogConfig)` and `AsyncVitalBuilder.logging(LogConfig)`. Two new resource clients are available: `CompendiumClient` (via `Vital.compendium()`) and `LabAccountClient` (via `Vital.labAccount()`), along with their async counterparts. Several enums have been expanded: `SONORA_QUEST` added to `AppointmentProvider`, `ClientFacingLabs`, and `Labs`; `CRL` added to `Labs`; `SAMSUNG_HEALTH` to `ManualProviders`; `TANDEM_SOURCE` to `PasswordProviders`; `APERO` and `PVERIFY` to `PayorCodeExternalProvider`; `CORRECTED` and `LAB_PROCESSING_BLOCKED` to `OrderLowLevelStatus`; and five new corrected/blocked statuses added to `OrderStatus`.
* The SDK now includes a new `CompendiumClient` (and async variant) with `search()` and `convert()` methods for interacting with the Compendium API. Request builder classes for activity, body, and aggregate resources now support `additionalProperty()` and `additionalProperties()` methods, allowing consumers to pass arbitrary extra fields when constructing requests. A new internal `SseEventParser` utility has also been added to support Server-Sent Events with discriminated union deserialization.
* The SDK now includes a `LabAccountClient` (and async variant) with a `getTeamLabAccounts()` method for retrieving team lab account information. Two new compendium request types — `SearchCompendiumBody` and `ConvertCompendiumBody` — are also available to support compendium search and conversion workflows. Several existing request builder classes gain `additionalProperty` / `additionalProperties` convenience methods for passing extra fields.
* The SDK now includes a new `RawLabAccountClient` for querying team lab accounts with optional filtering by `labAccountId` and `status`. The `CreateOrderRequestCompatible` request object gains a new optional `clinicalNotes` field. All lab test request builder classes now support `additionalProperty` and `additionalProperties` methods for passing arbitrary extra properties to requests.
* Request builder classes across the SDK now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to include arbitrary fields in request bodies. This enables forward-compatible usage with API attributes not yet formally modeled in the SDK.
* All request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods, allowing consumers to attach arbitrary extra fields to any request. This enables forward-compatibility with API parameters not yet formally modeled in the SDK.
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface and `Builder` implementations. This allows consumers to attach arbitrary extra parameters to requests across all Vitals resource types (cholesterol, glucose, heart rate, HRV, distance, floors climbed, ECG, and more).
* Vitals request builder classes now expose `additionalProperty(String key, Object value)` and `additionalProperties(Map<String, Object> additionalProperties)` methods on their `_FinalStage` interface, enabling consumers to attach arbitrary key-value pairs when constructing requests. This applies across all vitals request types including grouped and non-grouped variants.
* The SDK now exposes `additionalProperty` and `additionalProperties` builder methods on all request and type builder classes, allowing consumers to attach arbitrary extra fields. A new `AwakeningsValueMacroExpr` type has been added. `AppointmentBookingRequest` now supports an optional `appointmentNotes` field.
* The SDK now includes a new `CanonicalCandidate` type for working with lab marker canonical matching, containing LOINC/CPT metadata such as `loincSetHash`, `displayName`, `popularityScore`, and `confidence`. `ClientFacingAppointment` gains a new optional `appointmentNotes` field.
